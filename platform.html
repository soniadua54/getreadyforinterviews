What is the platform round
Read this first ▼
The most misunderstood round. Not pure system design. Not just coding. It is: LLD + practical backend thinking + code review — all set in Agoda's real product context.
What interviewers actually test
Three things evaluated simultaneously in 45 minutes:
1. Can you design a feature end-to-end? Given a booking/payment feature, can you think through the API, business logic, data model, and failure handling without being prompted at every step?
2. Can you read and improve existing code? They show you a code snippet — often buggy or poorly structured — and ask what's wrong and how you'd fix it.
3. Do you think in systems, not just functions? They want to see you mention caching, idempotency, error handling, observability — not just the happy path.
Real Agoda platform round prompts (reported)
  • Design a coupon/discount system for Agoda's booking flow
  • Given an existing hotel search backend, how would you improve performance for peak traffic?
  • Review this code snippet — what's wrong, what would you change?
  • Design an API for vendors to query their bookings on Agoda
  • Implement and explain a data structure (given abstract class, you implement)
The trap: If your resume lists Kubernetes, Spring Boot, Auth0 — they will ask you specifics. "I used it at work" is not an answer. Know one concrete thing you did with each technology.

LLD patterns you must know
Design patterns for backend systems ▼
Strategy pattern ★ Agoda
Use when behavior needs to swap at runtime. Agoda has 40+ payment methods across markets — this is exactly how you'd support them cleanly.
interface PaymentStrategy {
    boolean pay(double amount);
}

class WalletPayment implements PaymentStrategy {
    private double balance;
    public boolean pay(double amount) {
        if (balance < amount) return false;
        balance -= amount;
        return true;
    }
}

class BookingService {
    private PaymentStrategy strategy;

    public BookingService(PaymentStrategy strategy) {
        this.strategy = strategy;
    }

    public boolean checkout(double amount) {
        return strategy.pay(amount); // delegates to concrete strategy
    }
}
Observer pattern generic
When a booking is confirmed, multiple things need to happen: send email, update analytics, trigger loyalty points. Observer decouples them — the booking service doesn't know about any of them.
interface BookingObserver {
    void onBookingConfirmed(Booking booking);
}

class BookingService {
    private List<BookingObserver> observers = new ArrayList<>();

    public void addObserver(BookingObserver o) { observers.add(o); }

    public void confirmBooking(Booking b) {
        // ... save booking ...
        observers.forEach(o -> o.onBookingConfirmed(b)); // notify all
    }
}

class EmailNotifier implements BookingObserver {
    public void onBookingConfirmed(Booking b) {
        emailService.send(b.getUserEmail(), "Your booking is confirmed");
    }
}
Idempotency pattern ★ Agoda
The #1 correctness concern in payment and booking systems. If a network retry causes a double-charge — that's a critical incident. Idempotency keys prevent this.
public class BookingService {
    // In production: Redis with TTL. Here: in-memory for concept.
    private Map<String, BookingResult> processed = new ConcurrentHashMap<>();

    public BookingResult createBooking(String idempotencyKey, BookingRequest req) {
        // Return cached result if already processed
        if (processed.containsKey(idempotencyKey)) {
            return processed.get(idempotencyKey);
        }
        BookingResult result = processBooking(req);
        processed.put(idempotencyKey, result);
        return result;
    }
}
// Client sends: POST /bookings with header Idempotency-Key: uuid-v4
// On retry, same key → same result returned, no double booking
In production: store idempotency key in Redis with a 24h TTL. The key maps to the result JSON. On retry, Redis hit → return cached result, skip all business logic.
Repository pattern generic
Separates data access logic from business logic. Makes testing easier (mock the repository, not the DB). Standard in Spring Boot applications.
interface HotelRepository {
    Hotel findById(Long id);
    List<Hotel> findByCity(String city);
    Hotel save(Hotel hotel);
}

// Business logic doesn't know if data comes from MySQL, Redis, or mock
class HotelService {
    private final HotelRepository repo;

    public List<Hotel> searchHotels(String city) {
        return repo.findByCity(city);
    }
}
Circuit breaker pattern ★ Agoda
If payment gateway fails 5 times in 10 seconds, stop calling it for 30 seconds (fail fast). Prevents cascading failures. Java library: Resilience4j.
// Using Resilience4j
@CircuitBreaker(name = "paymentGateway", fallbackMethod = "fallback")
public PaymentResult charge(PaymentRequest req) {
    return externalPaymentGateway.charge(req);
}

public PaymentResult fallback(PaymentRequest req, Exception ex) {
    // Circuit is open — don't even try the gateway
    return PaymentResult.failed("Payment service temporarily unavailable");
}
// States: CLOSED (normal) → OPEN (failing) → HALF_OPEN (testing recovery)

Code review — what to look for
They will show you broken code ▼
When they give you a code snippet, don't just look at logic. Scan for these categories in order.
Bug 1: Null pointer
// BAD — NPE if hotel is null
public String getHotelName(Hotel hotel) {
    return hotel.getName();
}

// FIXED
public String getHotelName(Hotel hotel) {
    if (hotel == null) throw new IllegalArgumentException("hotel cannot be null");
    return hotel.getName();
}
Bug 2: Race condition ★ Agoda
// BAD — not atomic, race condition under concurrent access
private int bookingCount = 0;
public void increment() { bookingCount++; }

// FIXED — AtomicInteger is thread-safe
private AtomicInteger bookingCount = new AtomicInteger(0);
public void increment() { bookingCount.incrementAndGet(); }
Bug 3: Resource leak
// BAD — stream never closed if exception thrown
public void processFile(String path) throws IOException {
    FileInputStream fis = new FileInputStream(path);
    // process...
}

// FIXED — try-with-resources auto-closes
public void processFile(String path) throws IOException {
    try (FileInputStream fis = new FileInputStream(path)) {
        // process...
    }
}
Bug 4: Swallowed exception
// BAD — exception eaten silently, debugging impossible
try {
    processPayment(req);
} catch (Exception e) {
    // nothing here — you'll never know what failed
}

// FIXED — log and re-throw or handle specifically
try {
    processPayment(req);
} catch (PaymentGatewayException e) {
    log.error("Payment gateway error for bookingId={}", req.getBookingId(), e);
    throw new BookingException("Payment failed", e);
}
Bug 5: N+1 query ★ Agoda
// BAD — 1 query to get hotels + N queries for rooms = N+1 total
for (Hotel hotel : hotels) {
    List<Room> rooms = roomRepo.findByHotelId(hotel.getId());
}

// FIXED — 1 query to get all rooms for all hotel IDs
List<Long> hotelIds = hotels.stream().map(Hotel::getId).collect(toList());
Map<Long, List<Room>> roomsByHotel = roomRepo.findByHotelIdIn(hotelIds)
    .stream().collect(groupingBy(Room::getHotelId));
Bug 6: Wrong collection for concurrency
// BAD — HashMap is not thread-safe
private Map<String, Session> sessions = new HashMap<>();

// FIXED — ConcurrentHashMap for concurrent read/write
private Map<String, Session> sessions = new ConcurrentHashMap<>();
Code review checklist: NPE → race conditions → resource leaks → swallowed exceptions → N+1 queries → wrong concurrency primitives → missing input validation → hardcoded values.

Scenario: Design a coupon system ★ Agoda
Most likely platform round prompt. Think through every layer before coding anything.
Your answer structure
API layer: POST /coupons/validate {code, userId, bookingAmount} → {valid, discountAmount, finalAmount}

Business rules: one-time vs multi-use, expiry date, minimum spend, applicable hotel categories, per-user limit

Data model: Coupon(id, code, type, discountType, value, expiresAt, maxUses, usedCount), CouponUsage(couponId, userId, bookingId, usedAt)

Race condition: Two users redeem last use simultaneously → Redis atomic INCR on usedCount with check. If usedCount > maxUses → reject.
// Atomic coupon redemption with Redis
public boolean redeemCoupon(String code, Long userId) {
    String key = "coupon:uses:" + code;
    Long uses = redisTemplate.opsForValue().increment(key);
    if (uses > coupon.getMaxUses()) {
        redisTemplate.opsForValue().decrement(key); // rollback
        return false;
    }
    // persist usage to DB async
    return true;
}
Scenario: Improve hotel search performance ★ Agoda
Given: hotel search is slow during peak booking season. What do you do?
Step 1 — Profile first. What's slow? DB query? Serialization? Network? Don't guess.

Step 2 — Caching. Redis cache on search results keyed by (city, checkIn, checkOut, guestCount). TTL = 5 minutes. Cache invalidation: when hotel updates availability, evict relevant keys.

Step 3 — DB optimization. Composite index on (city, available, price). Read replica for search queries — don't hit the write primary.

Step 4 — Async logging. Don't let analytics logging block the API response. Push to Kafka, process offline.

Step 5 — Rate limiting. Bucket4j + Redis. 100 searches/minute per user. Prevents scraping from killing search infra.

Scenario: Implement LRU Cache must
Extremely common at Staff level. They may ask you to implement it live, or ask you to make it thread-safe.
class LRUCache {
    private final int capacity;
    private final LinkedHashMap<Integer, Integer> map;

    public LRUCache(int capacity) {
        this.capacity = capacity;
        // accessOrder=true maintains LRU order automatically
        map = new LinkedHashMap<>(capacity, 0.75f, true) {
            protected boolean removeEldestEntry(Map.Entry e) {
                return size() > capacity;
            }
        };
    }

    public int get(int key) {
        return map.getOrDefault(key, -1);
    }

    public void put(int key, int value) {
        map.put(key, value);
    }
}

// Thread-safe version: wrap get/put in synchronized blocks
// Or use Collections.synchronizedMap() wrapper

Java internals — Staff level depth
What they ask beyond the code ▼
Collections complexity — know cold
// ArrayList
get(i)     → O(1)   | add(end)  → O(1) amortized | add(mid) → O(n)
// LinkedList
get(i)     → O(n)   | add(end)  → O(1)            | add(front) → O(1)
// HashMap
get/put    → O(1) avg | O(n) worst (all keys same hash bucket)
// TreeMap (sorted keys)
get/put    → O(log n) | floorKey/ceilingKey → O(log n)
// PriorityQueue (min-heap)
peek       → O(1)   | poll/offer → O(log n)
// ArrayDeque (stack + queue)
push/pop/poll/offer → O(1)  ← use this, NOT Stack class
String vs StringBuilder vs StringBuffer
// String — immutable, thread-safe, creates new object on every concat
String s = "hello";
s = s + " world"; // creates NEW String object — O(n) each time!

// StringBuilder — mutable, NOT thread-safe, fast
StringBuilder sb = new StringBuilder();
sb.append("hello").append(" world"); // O(1) amortized

// StringBuffer — mutable, thread-safe, slow (synchronized)
// Use StringBuilder in interviews unless thread-safety is mentioned
Java memory model basics ★ Agoda
// volatile — ensures visibility across threads
// Does NOT guarantee atomicity
private volatile boolean running = true;
// One thread writes running=false, other thread sees it immediately

// synchronized — mutual exclusion + visibility
public synchronized void increment() { count++; } // only one thread at a time

// AtomicInteger — lock-free, better than synchronized for single variable
private AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet(); // atomic read-modify-write, no lock

// ReentrantLock — explicit lock with tryLock timeout
if (lock.tryLock(100, TimeUnit.MILLISECONDS)) {
    try { // do work } finally { lock.unlock(); }
}

Platform round prep checklist
Track your readiness ▼
Concepts to be able to explain verbally
  • Strategy, Observer, Repository patterns — give a booking domain example for each
  • Idempotency key pattern — why needed, how to implement with Redis
  • Circuit breaker — three states, Resilience4j usage, fallback method
  • LRU Cache implementation — both HashMap+DLL and LinkedHashMap approaches
  • Thread safety: volatile vs synchronized vs AtomicInteger — when to use each
  • @Transactional propagation types — explain REQUIRED vs REQUIRES_NEW
  • N+1 query — identify it in code, fix with batch query or JOIN
  • Redis caching with Spring — @Cacheable, cache key design, TTL, eviction
  • Rate limiting — token bucket algorithm, Bucket4j + Redis implementation
  • Cursor-based vs offset-based pagination — why cursor is better for large datasets
  • Kubernetes: know what you did — pod autoscaling, memory limits, CrashLoopBackoff debug
  • Prometheus alerting — what metrics you set up, what thresholds, what triggered
Resume defense rule: For every technology on your resume, prepare one concrete story: what you built, what broke, what you learned. They will ask. Vague answers here fail the platform round even if your code is perfect.
