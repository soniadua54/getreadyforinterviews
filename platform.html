What is the platform round
▼
The most misunderstood round. Not pure system design. Not just coding. It is LLD + practical backend thinking + code review — all in a real product context.
Three things evaluated simultaneously
1. End-to-end feature design — Given a feature, can you think through API, business logic, data model, and failure handling without being prompted?
2. Code review — They show you a snippet. What's wrong? What would you change?
3. Systems thinking — Do you think about caching, idempotency, error handling, observability? Or just the happy path?
Commonly reported platform prompts
  • Design a coupon/discount system for the booking flow
  • Given an existing hotel search backend — improve performance for peak traffic
  • Review this code snippet — what's wrong, what would you change?
  • Design an API for vendors to query their own booking data
  • Implement a data structure (given abstract class, fill in the implementation)
The resume trap: If your resume lists Kubernetes, Spring Boot, Redis — they will ask specifics. "I used it at work" is not enough. Have one concrete story per technology: what you built, what broke, what you fixed.

LLD design patterns
▼
Strategy pattern ★ Company
When behavior needs to swap at runtime. Classic example: multiple payment methods, each with different logic. Without strategy pattern you'd have a giant if-else chain that grows with every new payment method.
interface PaymentStrategy {
    boolean pay(double amount);
}
class WalletPayment implements PaymentStrategy {
    private double balance;
    public boolean pay(double amount) {
        if (balance < amount) return false;
        balance -= amount; return true;
    }
}
class BookingService {
    private PaymentStrategy strategy;
    public BookingService(PaymentStrategy s) { this.strategy = s; }
    public boolean checkout(double amount) {
        return strategy.pay(amount); // delegates to concrete impl
    }
}
New payment method? Just add a new class implementing PaymentStrategy. Zero changes to BookingService. This is Open/Closed principle.
Observer pattern generic
When one event triggers multiple actions. When a booking is confirmed: send email, update analytics, trigger loyalty points. Observer decouples the booking service from all these concerns.
interface BookingObserver { void onBookingConfirmed(Booking b); }

class BookingService {
    private List<BookingObserver> observers = new ArrayList<>();
    public void subscribe(BookingObserver o) { observers.add(o); }
    public void confirmBooking(Booking b) {
        // ... save to DB ...
        observers.forEach(o -> o.onBookingConfirmed(b)); // notify all
    }
}
class EmailNotifier implements BookingObserver {
    public void onBookingConfirmed(Booking b) {
        emailService.send(b.getUserEmail(), "Booking confirmed!");
    }
}
Idempotency pattern ★ Company
Network request fails. Client retries. Should the operation happen twice? For payments — absolutely not. Idempotency key ensures each logical operation executes exactly once regardless of retries.
public class BookingService {
    // Production: Redis with 24h TTL. This is the concept.
    private Map<String, BookingResult> processed = new ConcurrentHashMap<>();

    public BookingResult createBooking(String idempotencyKey, BookingRequest req) {
        // Already processed? Return cached result — no side effects
        if (processed.containsKey(idempotencyKey))
            return processed.get(idempotencyKey);

        BookingResult result = processBooking(req);
        processed.put(idempotencyKey, result);
        return result;
    }
}
// Client: POST /bookings  header: Idempotency-Key: {client-generated-uuid}
// On retry: same key → same result returned, no double booking
Production implementation: Redis SET key result EX 86400 NX. The NX flag means "only set if not exists" — atomically handles the check-and-set in one Redis command.
Circuit breaker ★ Company
If payment gateway fails 5 times in 10 seconds, stop calling it for 30 seconds. Fail fast. Prevents cascading failures where one slow service takes down your whole system.
// States: CLOSED (normal) → OPEN (stop calling) → HALF_OPEN (test recovery)
@CircuitBreaker(name="payment", fallbackMethod="fallback")
public PaymentResult charge(PaymentRequest req) {
    return externalGateway.charge(req);
}
public PaymentResult fallback(PaymentRequest req, Exception e) {
    return PaymentResult.failed("Payment temporarily unavailable, please retry");
}
// Library: Resilience4j (Java). Config: failureRateThreshold=50,
// waitDurationInOpenState=30s, permittedCallsInHalfOpenState=3
LRU Cache implementation must
Least Recently Used cache. When capacity is full, evict the item that was accessed longest ago. HashMap for O(1) lookup + Doubly Linked List for O(1) eviction order maintenance.
class LRUCache {
    private final int capacity;
    // LinkedHashMap with accessOrder=true maintains LRU order automatically
    private final LinkedHashMap<Integer, Integer> map;

    public LRUCache(int capacity) {
        this.capacity = capacity;
        map = new LinkedHashMap<>(capacity, 0.75f, true) {
            protected boolean removeEldestEntry(Map.Entry e) {
                return size() > capacity; // auto-evict when over capacity
            }
        };
    }
    public int get(int key) { return map.getOrDefault(key, -1); }
    public void put(int key, int val) { map.put(key, val); }
}
// Thread-safe version: wrap get/put in synchronized blocks

Code review — bug patterns to find
▼
When they show you code, scan in this order: NPE → race condition → resource leak → swallowed exception → N+1 query → wrong collection for concurrency.
Bug 1: Null pointer exception
// BAD — NPE if hotel is null
public String getHotelName(Hotel hotel) {
    return hotel.getName(); // throws NullPointerException
}
// FIXED
public String getHotelName(Hotel hotel) {
    Objects.requireNonNull(hotel, "hotel cannot be null");
    return hotel.getName();
}
// Or return Optional: Optional.ofNullable(hotel).map(Hotel::getName)
Bug 2: Race condition ★ Company
// BAD — not atomic. Read-modify-write is 3 operations, not 1
private int bookingCount = 0;
public void increment() { bookingCount++; } // RACE CONDITION

// FIXED — AtomicInteger uses CAS (compare-and-swap), lock-free
private AtomicInteger bookingCount = new AtomicInteger(0);
public void increment() { bookingCount.incrementAndGet(); }
Bug 3: Resource leak
// BAD — if exception thrown before close(), stream leaks
public void processFile(String path) throws IOException {
    FileInputStream fis = new FileInputStream(path);
    // process...
    fis.close(); // never reached on exception!
}
// FIXED — try-with-resources guarantees close() is always called
public void processFile(String path) throws IOException {
    try (FileInputStream fis = new FileInputStream(path)) {
        // process — fis.close() called automatically
    }
}
Bug 4: Swallowed exception
// BAD — exception disappears silently. You'll never know what failed.
try { processPayment(req); } catch (Exception e) { }

// FIXED — log with context, rethrow or handle specifically
try {
    processPayment(req);
} catch (PaymentException e) {
    log.error("Payment failed bookingId={} userId={}",
        req.getBookingId(), req.getUserId(), e);
    throw new BookingException("Payment failed", e);
}
Bug 5: N+1 query ★ Company
// BAD — 1 query to get hotels + N queries for rooms = N+1 total
for (Hotel h : hotels) {
    List<Room> rooms = roomRepo.findByHotelId(h.getId()); // N queries!
    h.setRooms(rooms);
}

// FIXED — 1 query for ALL rooms, then group by hotelId in memory
List<Long> ids = hotels.stream().map(Hotel::getId).collect(toList());
Map<Long, List<Room>> byHotel = roomRepo.findByHotelIdIn(ids) // 1 query
    .stream().collect(groupingBy(Room::getHotelId));
hotels.forEach(h -> h.setRooms(byHotel.getOrDefault(h.getId(), emptyList())));
Bug 6: HashMap used in concurrent context
// BAD — HashMap is NOT thread-safe. Can corrupt under concurrent access.
private Map<String, Session> sessions = new HashMap<>();

// FIXED — ConcurrentHashMap: thread-safe, no full-map lock
private Map<String, Session> sessions = new ConcurrentHashMap<>();
Know these cold: HashMap → not thread-safe. Hashtable → thread-safe but locks whole map (slow). ConcurrentHashMap → thread-safe, segment-level locking (fast). ArrayList → not thread-safe. CopyOnWriteArrayList → thread-safe reads, expensive writes.

Java internals — what gets asked
▼
Thread safety primitives ★ Company
// volatile — guarantees VISIBILITY, NOT atomicity
// Use for: flags, status variables read by multiple threads
private volatile boolean running = true;

// synchronized — mutual exclusion + visibility
// Use for: compound operations that must be atomic
public synchronized void transfer(double amount) {
    balance -= amount; // read + modify + write is atomic
}

// AtomicInteger — lock-free CAS operations
// Use for: counters, accumulators
private AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();     // atomic ++
count.compareAndSet(5, 10); // set to 10 only if currently 5

// ReentrantLock — explicit lock with timeout
// Use for: tryLock, interruptible waiting
if (lock.tryLock(100, TimeUnit.MILLISECONDS)) {
    try { /* critical section */ } finally { lock.unlock(); }
}
Collections complexity — know cold
ArrayList:     get O(1) | add-end O(1) amortized | add-mid O(n)
LinkedList:    get O(n) | add-front O(1)          | add-end O(1)
HashMap:       get/put O(1) avg  | worst O(n) all same bucket
TreeMap:       get/put O(log n)  | sorted iteration | floor/ceiling
PriorityQueue: peek O(1)  | poll/offer O(log n)
ArrayDeque:    push/pop/offer/poll all O(1)  ← use instead of Stack
Spring Boot annotations
@Transactional                      // ACID — all or nothing
@Transactional(readOnly=true)       // no write lock, faster reads
@Transactional(propagation=REQUIRES_NEW) // always creates new tx

@Cacheable("hotels")               // cache result, skip method on hit
@CacheEvict("hotels")              // remove from cache on update
@Async                              // run in separate thread pool
@Scheduled(cron="0 0 * * * *")   // run every hour

// PROPAGATION quick reference:
// REQUIRED (default): join existing tx or create new
// REQUIRES_NEW: always new tx, suspends outer if exists
// SUPPORTS: join if exists, proceed without tx if not
// MANDATORY: must have existing tx, throw if not
String vs StringBuilder
// String — immutable. Every concat creates a NEW object.
String s = "";
for (int i=0; i<1000; i++) s += i; // creates 1000 String objects! O(n²)

// StringBuilder — mutable, NOT thread-safe. Use in single thread.
StringBuilder sb = new StringBuilder();
for (int i=0; i<1000; i++) sb.append(i); // O(n), one object
String result = sb.toString();

// StringBuffer — mutable, thread-safe (synchronized). Slow.
// Use StringBuilder unless sharing across threads.

Scenario: coupon system design
▼
The most commonly reported platform prompt. Walk through every layer before writing any code.
API design
// Validate a coupon before applying it to a booking
POST /v1/coupons/validate
Body: { "code": "SUMMER20", "userId": 123, "bookingAmount": 150.00 }
Response: { "valid": true, "discountType": "PERCENT", "value": 20,
            "discountAmount": 30.00, "finalAmount": 120.00 }

// Apply coupon (idempotent — use booking ID as idempotency key)
POST /v1/coupons/redeem
Header: Idempotency-Key: booking-{bookingId}
Body: { "code": "SUMMER20", "userId": 123, "bookingId": 456 }
Data model
Coupon(id, code, discountType, value, minSpend, maxUses, usedCount,
       applicableCategories, expiresAt, createdAt)

CouponUsage(id, couponId, userId, bookingId, discountAmount, usedAt)

Indexes:
  UNIQUE(code) — fast lookup by code
  INDEX(userId) on CouponUsage — "has this user used this coupon?"
Race condition: two users redeem last use simultaneously
// Naive: SELECT usedCount, then UPDATE — race condition!
// Fixed: atomic increment with Redis, then validate

public boolean redeemCoupon(String code, Long userId, Long bookingId) {
    String key = "coupon:uses:" + code;

    // Atomic increment — only one thread gets each value
    Long uses = redis.increment(key);

    if (uses > coupon.getMaxUses()) {
        redis.decrement(key); // rollback — we over-counted
        throw new CouponExhaustedException("Coupon fully redeemed");
    }

    // Persist usage to DB (can be async via queue)
    couponUsageRepo.save(new CouponUsage(code, userId, bookingId));
    return true;
}
Why Redis INCR? Redis is single-threaded — INCR is atomic. No two threads can get the same value. If value > maxUses, we DECR to roll back and reject.
Business rules to mention
One-time vs multi-use coupons · Expiry date check · Minimum spend threshold · Per-user usage limit (CouponUsage table) · Category restrictions (only for flights, only for hotels) · Stackable vs non-stackable (can two coupons be applied at once?)

# Prometheus metric types:
# Counter:   only goes up (requests_total, errors_total)
# Gauge:     goes up and down (memory_bytes, active_connections)
# Histogram: samples in buckets (request_duration_seconds)
# Summary:   like histogram but pre-computed percentiles

# Spring Boot auto-exposes /actuator/prometheus with micrometer
# Key metrics to track for any service:
#
# http_requests_total{method, path, status}   → rate of requests
# http_request_duration_seconds{quantile=0.99} → p99 latency
# jvm_memory_used_bytes{area="heap"}           → heap usage
# hikaricp_connections{pool, state}            → DB pool saturation

# PromQL examples:
# Error rate: rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m])
# p99 latency: histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))
# Alert rule:
# - alert: HighErrorRate
#   expr: error_rate > 0.01  # 1% errors
#   for: 5m
#   annotations: { summary: "{{ $labels.service }} error rate > 1%" }
# ELK Stack: Elasticsearch + Logstash + Kibana
# Or: Loki (log storage) + Promtail (collector) + Grafana (visualise)

# Structured logging — always use JSON in production
# BAD:  log.info("User {} booked hotel {}", userId, hotelId);
# GOOD: log.info("Booking created", Map.of("userId", userId, "hotelId", hotelId, "traceId", traceId));

# Key log fields (every log entry should have these):
{
  "timestamp": "2026-07-09T09:00:00Z",
  "level":     "INFO",
  "service":   "booking-service",
  "traceId":   "abc123",     // links to distributed trace
  "userId":    "u456",
  "message":   "Booking confirmed",
  "durationMs": 142
}

# Log levels — use them correctly:
# ERROR: something is broken, needs immediate attention
# WARN:  something unexpected but system continues
# INFO:  normal operational events (booking created, payment processed)
# DEBUG: detailed info for debugging (don't log in production unless needed)
# NEVER log: passwords, card numbers, PII → legal liability
# SLI (Service Level Indicator): the metric you measure
# SLO (Service Level Objective): the target for that metric
# SLA (Service Level Agreement): the contractual commitment (usually weaker than SLO)
# Error budget: 100% - SLO = how much downtime/errors are allowed

# Example for booking API:
# SLI: % of booking requests with latency < 500ms
# SLO: 99.5% of requests under 500ms over a 30-day window
# Error budget: 0.5% = ~3.6 hours of total outage per month
# SLA: 99% (what we commit to customers — more lenient than internal SLO)

# When error budget is exhausted:
# - Freeze new feature releases
# - Focus all engineering on reliability
# - Post-incident review required

# Interview tip: "We use SLOs to make reliability a first-class engineering concern.
# When the error budget drops below 20%, we shift sprint priority from features to reliability work."

Production Debugging — Memory Leaks, Thread Dumps, High CPU
jstack · jmap · heap dumps · GC tuning · Linux tools
▼ 
# Scenario: Java service slowly consuming more memory, OOM after 6h
# 1. Check heap usage trend in Grafana (jvm_memory_used_bytes{area="heap"})
# 2. If growing steadily → likely memory leak

# Take heap dump:
jmap -dump:format=b,file=heap.hprof <pid>
# Or via JVM flag: -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/

# Analyse with Eclipse MAT or VisualVM:
# "Leak Suspects" report → shows objects accumulating
# Common causes: static caches growing unbounded, listeners not deregistered,
#                ThreadLocal not cleaned up, Hibernate session not closed

# Scenario: service high CPU, requests timing out
# 1. Identify PID: ps aux | grep java | sort -k3 -r
# 2. Find hot threads: top -H -p <pid>  (H = show threads)
# 3. Take thread dump: jstack <pid> > thread.txt
# 4. In thread dump: RUNNABLE threads in tight loop = CPU hog
# 5. Common cause: infinite loop, regex catastrophic backtracking, GC thrashing

# GC thrashing: GC spending > 10% of time → not enough heap
# Fix: -Xmx4g (increase max heap) or find memory leak
# View GC: -Xlog:gc*:file=/tmp/gc.log:time,uptime:filecount=5,filesize=20m

# Linux tools for production debugging:
# top / htop:  CPU and memory per process
# iostat -x:   disk I/O wait (high await = disk bottleneck)  
# netstat -an: open connections (too many TIME_WAIT = connection pool problem)
# strace -p:   system calls (advanced — use sparingly in prod)
# lsof -p:     open file handles (leaking file descriptors → FD limit hit)
