Spring Boot Dockerfile — Basic
Every instruction creates a new layer. Order matters: put things that change less often at the top, things that change often (your code) at the bottom.
# FROM — base image (always pin the version, never use :latest in prod)
FROM eclipse-temurin:17-jre-alpine

# WORKDIR — all subsequent commands run from here
WORKDIR /app

# COPY — copy the built JAR from your host into the container
# The dot at the end means: copy to WORKDIR (/app)
COPY target/hotel-service-1.0.jar app.jar

# EXPOSE — document which port the app listens on (doesn't actually open it)
EXPOSE 8080

# ENV — environment variables available at runtime
ENV SPRING_PROFILES_ACTIVE=prod

# ENTRYPOINT — the command to run. Prefer ENTRYPOINT over CMD for apps.
ENTRYPOINT ["java", "-XX:+UseContainerSupport", "-jar", "app.jar"]
# -XX:+UseContainerSupport: makes JVM respect container memory limits (critical!)
Common mistake: Using FROM openjdk:17 (1.1 GB!) vs eclipse-temurin:17-jre-alpine (~175 MB). Always use JRE (not JDK) in production — you only need to run, not compile. Alpine base cuts image size dramatically.
Multi-Stage Build — Compile in one stage, run in another
Multi-stage builds solve the problem: your build tools (Maven, JDK) are needed to compile but you don't want them in your production image. Build in a full JDK image, copy only the JAR into a minimal JRE image.
# ─── STAGE 1: Build ───────────────────────────────
# Full JDK with Maven — only used during build, not in final image
FROM eclipse-temurin:17-jdk-alpine AS builder

WORKDIR /build

# Copy pom.xml first — so Maven dependency layer is cached
# (dependencies only re-downloaded when pom.xml changes)
COPY pom.xml .
RUN mvn dependency:go-offline -q

# Now copy source and build
COPY src/ src/
RUN mvn package -DskipTests -q

# ─── STAGE 2: Run ─────────────────────────────────
# Minimal JRE — this is the final image (no Maven, no JDK, no source code)
FROM eclipse-temurin:17-jre-alpine

WORKDIR /app

# Create non-root user — security best practice
RUN addgroup -S spring && adduser -S spring -G spring
USER spring:spring

# Copy ONLY the JAR from the builder stage
COPY --from=builder /build/target/hotel-service-*.jar app.jar

EXPOSE 8080
ENTRYPOINT ["java", "-XX:+UseContainerSupport", "-Xms256m", "-Xmx512m", "-jar", "app.jar"]

# Result: ~175 MB instead of 500+ MB with full JDK
#         No build tools exposed in final image
#         Non-root user = less attack surface
💡 Interview answer
Multi-stage builds solve two problems: (1) image size — final image only has runtime deps, not build tools; (2) security — your source code and build tooling never make it into the production image. At UKG we'd use this to keep Spring Boot images under 200 MB vs the 500+ MB you'd get with a naive build.
Layer Caching — Why Order of Instructions Matters
Docker caches each layer. If a layer changes, all layers below it are invalidated. This is why you put slow, rarely-changing steps first (dependency download) and fast, frequently-changing steps last (your code).
# ❌ WRONG — cache busted every time any source file changes
COPY . .                 # copies everything including source
RUN mvn dependency:resolve  # re-downloads ALL deps every build

# ✅ RIGHT — Maven dependencies cached separately from source
COPY pom.xml .           # Step 1: copy only manifest
RUN mvn dependency:go-offline  # Step 2: cache deps (only reruns if pom.xml changes)
COPY src/ src/           # Step 3: copy source (only this layer + below rebuild)
RUN mvn package          # Step 4: compile

# Same principle for Node.js:
COPY package*.json .     # copy package.json first
RUN npm ci               # install deps — cached until package.json changes
COPY . .                 # copy source after — deps already cached

# Useful commands:
# docker build -t hotel-service .       Build image
# docker build --no-cache -t hotel .    Force rebuild all layers
# docker history hotel-service          Show each layer's size

Docker Compose — Local Dev Environment
Run your full stack locally: Spring Boot + MongoDB + Redis + Kafka in one command
Intermediate ▼
docker-compose.yml — Agoda hotel booking local stack
# docker-compose.yml — local development environment
# Run: docker compose up -d
# Stop: docker compose down (add -v to also remove volumes)

version: '3.9'

services:
  # ── Your Spring Boot app ──────────────────────────
  hotel-service:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "8080:8080"         # host:container
    environment:
      SPRING_DATA_MONGODB_URI: mongodb://mongo:27017/hotels
      SPRING_REDIS_HOST: redis
      SPRING_KAFKA_BOOTSTRAP_SERVERS: kafka:9092
      SPRING_PROFILES_ACTIVE: local
    depends_on:
      mongo:
        condition: service_healthy   # wait for healthcheck before starting
      redis:
        condition: service_healthy
    volumes:
      - ./logs:/app/logs           # persist logs to host
    networks:
      - hotel-net

  # ── MongoDB ───────────────────────────────────────
  mongo:
    image: mongo:7
    ports:
      - "27017:27017"
    volumes:
      - mongo_data:/data/db        # named volume — data persists across restarts
    environment:
      MONGO_INITDB_ROOT_USERNAME: admin
      MONGO_INITDB_ROOT_PASSWORD: "${MONGO_PASSWORD}"  # from .env file
    healthcheck:
      test: ["CMD", "mongosh", "--eval", "db.runCommand('ping').ok"]
      interval: 10s
      timeout: 5s
      retries: 3
    networks:
      - hotel-net

  # ── Redis ─────────────────────────────────────────
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    command: redis-server --requirepass "${REDIS_PASSWORD}"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      retries: 3
    networks:
      - hotel-net

  # ── Kafka + Zookeeper ─────────────────────────────
  zookeeper:
    image: confluentinc/cp-zookeeper:7.4
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181
    networks:
      - hotel-net

  kafka:
    image: confluentinc/cp-kafka:7.4
    ports:
      - "9092:9092"
    environment:
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092
      KAFKA_AUTO_CREATE_TOPICS_ENABLE: "true"
    depends_on:
      - zookeeper
    networks:
      - hotel-net

volumes:
  mongo_data:              # named volume — survives compose down

networks:
  hotel-net:
    driver: bridge         # default; containers on same network can reach each other by service name
Key concept: Services on the same Docker network reach each other by service name. Your Spring Boot app connects to mongodb://mongo:27017 — not localhost:27017. Docker's internal DNS resolves mongo to the MongoDB container's IP.

Networking
Security hardening
Debug commands
# Docker network modes
# bridge (default) — isolated network, port mapping to host
docker network create hotel-net
docker run --network hotel-net hotel-service

# host — container shares host's network stack (no isolation)
# Use for high-performance, but NO port mapping needed
docker run --network host hotel-service

# none — no networking (offline processing jobs)
docker run --network none data-processor

# Inspect networks
docker network ls
docker network inspect hotel-net       # see which containers are connected
docker exec hotel-service ping mongo   # verify container-to-container connectivity

# Port mapping (-p host:container)
docker run -p 8080:8080 hotel-service  # accessible at localhost:8080
docker run -p 127.0.0.1:8080:8080 hotel-service  # only from localhost (more secure)
# Security checklist for production Docker images

# 1. Non-root user — default is root (security risk)
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser           # run as non-root

# 2. Read-only filesystem (prevent runtime writes)
docker run --read-only hotel-service
# Mount specific write paths:
docker run --read-only -v /tmp:/tmp hotel-service

# 3. Drop Linux capabilities
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE hotel-service
# Principle of least privilege: only add what's needed

# 4. No privileged containers
# NEVER: docker run --privileged ... (gives full host access)

# 5. Resource limits (prevent container starving host)
docker run --memory="512m" --cpus="1.0" hotel-service

# 6. Secrets — don't bake into image
# BAD: ENV DB_PASSWORD=mysecret  ← visible in docker inspect
# GOOD: Use Docker secrets or pass at runtime:
docker run -e DB_PASSWORD="$(vault kv get -field=password secret/db)" hotel-service

# 7. Scan image for vulnerabilities
docker scout cves hotel-service:latest    # Docker Scout (built-in)
trivy image hotel-service:latest           # Trivy (open source, CI/CD friendly)

# 8. Don't use :latest in production — pin versions
# BAD:  FROM openjdk:latest
# GOOD: FROM eclipse-temurin:17.0.11_9-jre-alpine
# Essential Docker debug commands

# See running containers
docker ps
docker ps -a                            # include stopped containers

# See logs
docker logs hotel-service               # all logs
docker logs -f hotel-service            # follow (like tail -f)
docker logs --tail 100 hotel-service    # last 100 lines

# Shell into running container
docker exec -it hotel-service /bin/sh   # Alpine: sh not bash
docker exec -it hotel-service bash      # if bash is available

# Shell into container without starting the app (debug startup issues)
docker run -it --entrypoint /bin/sh hotel-service

# Resource usage
docker stats                            # live CPU/memory/network usage
docker stats hotel-service --no-stream  # one-time snapshot

# Inspect container config
docker inspect hotel-service            # full JSON: ports, env, mounts, network
docker inspect hotel-service | jq '.[0].HostConfig.Memory'  # memory limit

# Cleanup
docker system prune                     # remove stopped containers, unused images
docker system prune -a                  # also remove images not used by any container
docker volume prune                     # remove unused volumes
docker image ls                         # see image sizes

  • Docker Compose networking — how does hotel-service reach mongo:27017?
  • depends_on with healthcheck — why not just depends_on without condition?
  • Why run containers as non-root? What risk does root in container create?
  • How do you pass secrets to a container without baking them into the image?
  • Bridge vs host network — when would you use host networking?
  • Named volumes vs bind mounts — what persists after docker compose down?
  • docker stats — how do you see if a container is memory-constrained?
  • How do you debug a container that exits immediately on startup?
  • docker system prune -a — what does it remove, what stays? (running containers stay)
