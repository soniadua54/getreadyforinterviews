System Design Round
Hotel booking · Search · Payment · Notifications · Rate limiter · CAP · CQRS · Caching · Sharding · Observability
System design is the round where Staff-level thinking is most visible. Senior engineers describe what to build. Staff engineers describe trade-offs, failure modes, and why. Every answer here has a structure: clarify → estimate → HLD → data model → deep dive on the hardest part. Know the hotel booking system cold — it covers 80% of what comes up.
📐
The framework — use every time
▼
Every system design answer follows the same 5-step structure. Interviewers expect you to drive it. If they're asking "what about scale?" — you already lost points.
5-step template
Step 1 — Clarify (2 min): Functional requirements (what must it do?) and non-functional (scale: DAU, req/sec; latency: <200ms?; consistency: strong or eventual OK?; availability: 99.9% or 99.99%?)

Step 2 — Estimate (2 min): DAU × requests/user = req/day → ÷86400 = req/sec. Storage: record_size × writes/day × retention_days. Read:write ratio. Tells you where bottlenecks will be.

Step 3 — HLD (5 min): Draw boxes first. Client → LB → API Gateway → Services → DB/Cache → Queue. No details yet — get the skeleton agreed on.

Step 4 — Data model (5 min): Key tables, primary keys, important indexes. Don't skip — candidates get eliminated here most often.

Step 5 — Deep dive on hardest part (15 min): Usually: concurrency (double booking), caching strategy, failure handling. Staff-level is differentiated from Senior here.
Senior vs Staff level answers
Senior: "I'd use a unique constraint on (roomId, checkIn) to prevent double bookings."

Staff: "At low scale: DB unique constraint. At high concurrency: Redis distributed lock (SETNX with TTL) as first gate, optimistic locking with version column in DB as second, unique constraint as final backstop. Three layers, each handles a different load range, each with different trade-offs I'm happy to walk through."
Always give trade-offs. "Use Kafka" is not an answer. "Kafka for at-least-once delivery with replay capability, vs SQS which is simpler to operate but loses ordering — for this use case I'd choose X because Y" is a Staff-level answer.

Core concepts — know these cold
▼
CAP theorem
Distributed systems can guarantee only 2 of 3: Consistency, Availability, Partition tolerance. Since network partitions always happen — you choose CP or AP.

CP — Consistency + Partition

Return error if data may be stale. Use for: payments, bookings, inventory. You'd rather say "room unavailable" than double-book.

AP — Availability + Partition

Return potentially stale data rather than error. Use for: hotel search results, review counts, recommendations. Stale for a few seconds is fine.

Caching strategies
Cache-aside (lazy):  App checks cache → miss → load DB → store cache.
                     Most common. App owns cache logic. TTL-based eviction.

Write-through:       Write to cache AND DB together. Always consistent.
                     Slower writes. Good when reads immediately follow writes.

Write-behind:        Write cache only. Async flush to DB. Fastest writes.
                     Risk: data loss if cache crashes before flush.

Cache key design:    search:{city}:{checkIn}:{checkOut}:{guests}
TTL strategy:        Availability data: 5 min (changes frequently)
                     Hotel metadata: 24h (changes rarely)
When to use which database
PostgreSQL / MySQL: Transactions, ACID guarantees, complex joins. Use for: bookings, payments, user accounts. Correctness over performance.

MongoDB / DynamoDB: Flexible schema, horizontal scale. Use for: hotel catalog (different hotels have different attributes), user preferences, product catalog.

Redis: Cache, sessions, rate limiting, distributed locks, leaderboards. Sub-ms reads. Data must fit in memory.

Elasticsearch: Full-text search, faceted filtering, geo queries. Use for: hotel search by name/location/amenities. NOT for transactional data.

Cassandra: Write-heavy, time-series, multi-region. Use for: event logs, analytics, IoT data. Wide column, no joins.
Message queues — when to use which
Kafka:      High throughput (1M msg/sec). Durable. Replayable.
            Ordered within partition. Consumer groups for parallel processing.
            Use for: event streaming, audit logs, notification fanout, analytics.

SQS:        Managed, auto-scales, at-least-once delivery.
            No ordering guarantee (FIFO queue variant exists).
            Use for: task queues, async jobs, simple decoupling.

RabbitMQ:   Flexible routing (exchanges), dead-letter queues, ack-based.
            Use for: task distribution, priority queues, complex routing.
Numbers to know
L1 cache:          0.5 ns   | RAM:           100 ns
SSD random read:   150 µs   | Network same DC: 500 µs
Network cross-DC:  150 ms

1M  req/day  →   ~12 req/sec
10M req/day  →  ~115 req/sec
1B  req/day  → ~11,600 req/sec

Single PostgreSQL write: ~1,000 QPS | read replica: ~10,000 QPS
Redis:                   ~100,000 ops/sec
Kafka (cluster):         ~1,000,000 msg/sec

Design: hotel booking system ★ Company
▼
The core travel product. Very likely to appear in a system design round. Know every layer and the concurrency solution cold.
Requirements to state upfront
Functional: Search hotels by city/dates/guests. View room availability. Book a room. Cancel booking. View booking history.

Non-functional: 10M DAU. 100K concurrent searches. 1K bookings/sec at peak. Search latency <200ms. No double booking (strong consistency on inventory). 99.99% availability.
Data model
Hotel     (id, name, city, country, lat, lng, star_rating, amenities[])
Room      (id, hotel_id, type, max_guests, price_per_night, total_rooms)
Inventory (room_id, date, available_count, booked_count)   ← key table
Booking   (id, user_id, room_id, check_in, check_out, status,
           total_price, idempotency_key, created_at)
           status: PENDING | CONFIRMED | CANCELLED | REFUNDED
Payment   (id, booking_id, amount, currency, status, gateway_txn_id)

Indexes:
  Hotel search:   INDEX(city, star_rating) + GEO INDEX(lat, lng)
  Availability:   INDEX(room_id, date) on Inventory table
  User history:   INDEX(user_id, created_at DESC) on Booking
Preventing double booking — three layers
Problem: 500 users request the last room simultaneously. All read available_count=1. All proceed to book. Without protection: 500 confirmed bookings for 1 room.
-- Layer 1: Redis distributed lock (fast gate, rejects most)
SET lock:room:123:2026-07-01 {userId} NX PX 30000
-- NX = only set if not exists. PX = auto-expire in 30s.
-- Returns nil if key exists → another user holds lock → reject

-- Layer 2: Optimistic locking in DB (handles what Redis misses)
UPDATE inventory
SET available_count = available_count - 1, version = version + 1
WHERE room_id=? AND date=? AND available_count > 0 AND version=?;
-- 0 rows affected = concurrent update → return "room unavailable"

-- Layer 3: DB unique constraint (final backstop)
-- UNIQUE INDEX on Booking(room_id, check_in, check_out)
-- Catches anything that slipped through layers 1 and 2
Production answer: Redis first (fast rejection), optimistic lock in DB (handles race), unique constraint (backstop). Three layers, each handles a different failure mode.
High-level architecture
Client → CDN → Load Balancer
                    ↓
              API Gateway (auth, rate limit, routing)
                    ↓
     ┌──────────────┼──────────────┐
Search Service  Booking Service  User Service
     ↓               ↓
Elasticsearch   PostgreSQL (primary write)
Redis (cache)   Redis (distributed lock)
                Payment Service → External Gateway
                Kafka → Notification Service → Email/SMS/Push

Design: hotel search system ★ Company
▼
Why not just SQL?
Full-text search in SQL is slow. Geo-radius queries need specialized indexes. Faceted filtering ("show me count of results per star rating") is expensive. Multi-signal ranking doesn't fit relational model.
Answer: Elasticsearch for search (text, geo, filtering, ranking) + Redis bitmap for real-time availability + PostgreSQL as source of truth for hotel data.
Search flow
User query → Search Service
  (1) Elasticsearch: query hotel metadata, amenities, location
      Returns: ranked list of hotel IDs
  (2) Redis availability filter: for each hotel ID, check
      available_count for requested dates
      Returns: subset with available rooms
  (3) Merge scores + availability, paginate
  (4) Cache result in Redis (key=city:checkIn:checkOut:filters, TTL=5min)
  → Return to user

Elasticsearch index document:
{
  hotel_id, name, city, country,
  location: { lat: 13.75, lon: 100.52 },  // geo_point type
  star_rating: 4, review_score: 8.7, price_min: 80,
  amenities: ["wifi", "pool", "gym", "parking"]  // keyword array
}
Keeping the index fresh
Hotel metadata updated → Kafka "hotel.updated"
                              ↓
                     Indexer Service (Kafka consumer)
                              ↓
                     Elasticsearch partial update (O(1))

Availability changes → NOT indexed in Elasticsearch
                    → Too frequent (every booking changes it)
                    → Stored in Redis bitmaps per (room_id, date)
                    → Applied as post-search filter step

Design: payment & wallet system ★ Company
▼
Core constraints
No double charges. No lost payments. Full audit trail. Cross-currency support. PCI DSS compliance. Exactly-once processing even on retries.
Data model
Wallet      (id, user_id, currency, balance, version)  // version = optimistic lock
Transaction (id, wallet_id, type, amount, currency, status,
             reference_id, idempotency_key, created_at)
             type: CREDIT | DEBIT | REFUND | FEE
             status: PENDING | COMPLETED | FAILED | REVERSED
Outbox pattern — exactly-once event publishing
Problem: if you save payment to DB AND publish event to Kafka in two separate operations, one can succeed and the other fail → inconsistent state forever.
// Solution: outbox pattern — atomic write to DB + outbox table
BEGIN TRANSACTION;
  UPDATE wallet SET balance = balance - amount WHERE id = ?;
  INSERT INTO outbox (event_type, payload, created_at)
         VALUES ('PAYMENT_PROCESSED', '{...}', NOW());
COMMIT;
// Separate poller reads outbox → publishes to Kafka → deletes row
// If publish fails → retry from outbox. Never lose an event.
// Payment and event publishing are always consistent — same DB transaction.
Refund flow
Refund = new CREDIT transaction (NOT a reversal).
This preserves the full audit trail — you can always see the debit and credit.

Request validated → create REFUND Transaction (PENDING)
→ call payment gateway refund API
→ success: mark COMPLETED, credit wallet
→ failure: mark FAILED, push to manual review queue, alert on-call
→ publish RefundCompleted event → Kafka → email notification

Design: notification system ★ Company
▼
Key insight: never block booking on notification
A slow email provider should not make every booking slow. Decouple completely with a message queue. The booking service just publishes an event — it doesn't know or care who receives it.
Booking Service → Kafka topic "booking.events"
                         ↓
                Notification Service (Kafka consumer)
                reads event → routes by channel preference
                         ↓
          ┌──────────────┼───────────────┐
     Email (SendGrid)  SMS (Twilio)  Push (FCM/APNs)
Reliability — what if a channel is down?
Kafka retains messages until consumer commits offset.
Email service down → messages wait in Kafka partition.
Service recovers → resumes from last committed offset.

Retry with exponential backoff:
  Attempt 1: wait 10s → attempt 2: 30s → attempt 3: 90s
  After 3 failures → move to Dead Letter Queue (DLQ) topic
  Alert on-call → manual review

User preference table:
NotificationPref(user_id, channel, event_type, enabled, quiet_hours_start)
Before sending: check — user opted out of SMS? Skip. In quiet hours? Defer push.

CAP Theorem & Consistency Models
CP vs AP · Strong vs Eventual · Read-your-writes · Linearizability
▼
CAP Theorem — what it actually means in practice
CAP says: in a distributed system with a network partition, you must choose between Consistency (every read sees latest write) and Availability (every request gets a response, may be stale). You can't have both. Without partition: you get both. The question is what happens when a partition occurs.
SystemChoosesWhat happens on partitionUse when
MySQL (primary-replica)CPReplica goes read-only if can't reach primaryBookings, payments — stale data is dangerous
Cassandra / DynamoDBAPStill returns data, may be staleShopping carts, user preferences — stale OK
Redis SentinelCPNo writes until leader electedDistributed locks, rate limiting
Zookeeper / etcdCPMajority quorum required for writesConfig, leader election
Consistency models — weakest to strongest
// Eventual consistency — Cassandra, DynamoDB default
// Writes propagate asynchronously. Reads may see stale data.
// Good for: shopping carts, likes, view counts, recommendations

// Read-your-writes (session consistency)
// You always see your own writes. Others may see stale.
// Implement: route user's reads to same replica, or use sticky sessions

// Monotonic read consistency  
// If you read a value, future reads never return older values.
// Good for: news feed (don't show post then hide it)

// Strong consistency (linearizability)
// All reads see the most recent committed write.
// Single-node or consensus protocol (Raft, Paxos).
// Good for: bank balances, inventory counts, booking availability

// Interview answer for booking system:
// Availability check (search): eventual consistency OK, stale by seconds fine
// Booking write: strong consistency required — use SELECT FOR UPDATE or OCC
// Payment: strong consistency, idempotency key to prevent double charge

CQRS & Event Sourcing
Command Query Responsibility Segregation · Event store · Projections
▼
CQRS
Event Sourcing
Saga Pattern
CQRS: Separate the read model (Query) from the write model (Command). Write side handles commands and emits events. Read side maintains denormalized projections optimised for queries. Solves the impedance mismatch between write-optimised (normalised) and read-optimised (denormalised) data models.
// Command side — normalised, consistent
BookingCommand → BookingAggregate → BookingEvent → EventStore

// Query side — denormalised, fast reads
EventHandler → BookingReadModel (Elasticsearch/Redis)
               ↓
               GET /hotels/{id}/availability → O(1) from read model

// Why CQRS for hotel search:
// Write: 1 booking → affects 1 inventory row (simple write)
// Read: search → join hotel + rooms + prices + reviews + availability (complex)
// Separate models → each optimised independently
When NOT to use CQRS: Simple CRUD apps. CQRS adds eventual consistency between command and query sides — you have to handle the lag. Only worth it when read and write patterns are fundamentally different in shape.
Event Sourcing: Instead of storing current state, store the sequence of events that led to it. Current state = replay of all events. Gives you full audit trail, time travel, and event-driven integration for free.
// Traditional: store current state
bookings table: id=1, status=CONFIRMED, price=200

// Event sourcing: store events
events table:
  {type: BOOKING_CREATED,    bookingId: 1, price: 200, ts: T1}
  {type: PRICE_ADJUSTED,     bookingId: 1, price: 180, ts: T2}
  {type: BOOKING_CONFIRMED,  bookingId: 1, ts: T3}

// Current state = fold over events
// Pros: full audit, easy debugging, replay to fix bugs
// Cons: eventual consistency in projections, snapshot needed for large event streams
Saga pattern: Manages distributed transactions across microservices without a 2-phase commit. Each step in the saga publishes an event. If a later step fails, compensating transactions undo previous steps.
// Booking saga: BookingService → InventoryService → PaymentService → NotificationService

Step 1: BookingService creates booking (status=PENDING) → publishes BOOKING_CREATED
Step 2: InventoryService reserves room → publishes INVENTORY_RESERVED  
Step 3: PaymentService charges card → publishes PAYMENT_PROCESSED
Step 4: BookingService confirms → publishes BOOKING_CONFIRMED
Step 5: NotificationService sends email

// If Step 3 fails (payment declined):
Compensation 2: InventoryService releases reservation → publishes INVENTORY_RELEASED
Compensation 1: BookingService cancels → publishes BOOKING_CANCELLED

// Choreography vs Orchestration:
// Choreography: each service listens to events and decides what to do (decoupled, hard to trace)
// Orchestration: a saga orchestrator sends commands and waits for responses (clearer, single failure point)

Database Sharding & Replication
Horizontal scaling · Sharding strategies · Read replicas · Replication lag
▼ 
// REPLICATION: primary handles writes, replicas handle reads
// Primary → binlog → replicas (async by default)
// Replication lag: replica may be seconds behind primary
// Problem: user writes booking, immediately reads → reads stale replica (booking not there)
// Fix: route writes AND immediate post-write reads to primary (sticky reads)

// SHARDING: split data horizontally across multiple DB nodes
// Each shard holds a subset of rows

// Sharding strategy 1: Range sharding
// hotel_id 1-10,000 → Shard A, 10,001-20,000 → Shard B
// Pro: range queries easy. Con: hotspots (new hotels all go to latest shard)

// Sharding strategy 2: Hash sharding (most common)
// shard = hash(hotel_id) % num_shards
// Pro: even distribution. Con: range queries need all shards, resharding is painful

// Sharding strategy 3: Directory (lookup service)
// A routing service maps hotel_id → shard_id
// Pro: flexible. Con: routing service is a bottleneck/SPOF

// WHAT TO SHARD ON for hotel booking:
// hotel_id: hotel-specific queries fast, cross-hotel aggregations slow
// user_id: user's bookings fast, availability check needs broadcast
// Common pattern: shard on hotel_id, denormalize user's bookings to separate read store

// Cross-shard transactions — the big problem
// If booking spans two shards (user on shard A, hotel on shard B):
// Option 1: two-phase commit (2PC) — slow, blocking
// Option 2: saga — eventually consistent compensations
// Option 3: co-locate related data on same shard (hotel + its bookings together)

Distributed Tracing & Observability
OpenTelemetry · Trace ID propagation · Metrics vs Logs vs Traces
▼ 
// THE OBSERVABILITY TRIAD
// Metrics:  what is happening right now (numbers) → Prometheus + Grafana
// Logs:     what happened, with context (text) → ELK stack / Loki
// Traces:   how a request flows through services → Jaeger / Zipkin / OpenTelemetry

// DISTRIBUTED TRACING: a single user request spans multiple services
// TraceId: unique ID for the entire request journey  
// SpanId: ID for one service's portion of the work
// ParentSpanId: links child span to parent

// HTTP header propagation (OpenTelemetry W3C format):
// traceparent: 00-{traceId}-{spanId}-{flags}

// Flow for hotel booking:
// Browser → API Gateway → BookingService → InventoryService → PaymentService
//              traceId=abc123  spanId=1        spanId=2           spanId=3
//              (root span)     parent=1        parent=1           parent=2

// Jaeger query: find trace abc123 → see full waterfall, find which service was slow

// KEY METRICS to track (USE method):
// Utilization: CPU%, memory%, DB connection pool usage
// Saturation:  queue depth, request backlog  
// Errors:      5xx rate, timeout rate, payment failure rate

// ALERTING: alert on symptoms not causes
// BAD:  alert when CPU > 80% (might be fine)
// GOOD: alert when p99 latency > 500ms OR error rate > 0.1%
