The framework — use every time
Structure every answer with this ▼
Every system design answer follows the same structure. Interviewers expect you to drive it. If they have to ask "but what about scale?" — you already failed that part.
The 5-step template
1. Clarify requirements (2 min)
Functional: what the system must do (book a hotel, search, pay)
Non-functional: scale (req/sec, users), latency target (<200ms?), consistency (strong or eventual ok?), availability (99.9% or 99.99%?)

2. Capacity estimation (2 min)
DAU × requests/user = req/day → req/sec. Storage: (record size × writes/day × days). Read:write ratio. This tells you where bottlenecks are.

3. High-level design (5 min)
Draw boxes first: Client → Load Balancer → API Gateway → Services → DB/Cache → Message Queue. No details yet.

4. Data model (5 min)
Key tables/documents, primary key, important indexes, relationships. Don't skip this — it's where most candidates get caught.

5. Deep dive on the hardest part (15 min)
Usually: concurrency (double booking), caching strategy, failure handling, consistency guarantees. This is where Staff-level is differentiated from Senior.
Staff-level vs Senior-level answers
Senior answer: "I'd use a relational DB with a unique constraint on (roomId, checkIn) to prevent double bookings."

Staff answer: "For the double-booking problem: at low scale, a DB-level unique constraint works. At Agoda's scale with high concurrent writes, I'd add a Redis distributed lock (SETNX with 30s TTL) as the first gate, then do optimistic locking with a version column in the DB as a second guard. If version mismatch on update → retry. For the ~0.1% race that passes both — the unique constraint at DB level is the final backstop. Three layers, each handles a different load range."
The pattern: Always give trade-offs. Never just say "use Kafka" — say "Kafka for at-least-once delivery with offset management, but adds latency and operational complexity. SQS is simpler if you don't need ordering. Given this use case I'd choose X because Y."

Core concepts — know these cold
Building blocks for every design ▼
CAP theorem
A distributed system can only guarantee 2 of 3: Consistency, Availability, Partition tolerance. Since network partitions happen — you choose CP or AP.
Booking/payment → CP. You'd rather show "room unavailable" than double-book. Correctness over availability.

Hotel search rankings, review counts → AP. Stale data for a few seconds is fine. Availability wins.
Consistency models
Strong consistency:    Read always sees the latest write. Expensive. (RDBMS default)
Eventual consistency: Read may see stale data, but will converge. Fast. (DynamoDB, Cassandra)
Read-your-writes:     After you write, you always read your own write.
Monotonic reads:      Once you read value X, you never read older value Y.
Caching strategies
Cache-aside (lazy):   App checks cache first. Miss → load from DB → populate cache.
                      Most common. App owns cache management.

Write-through:        Write to cache AND DB simultaneously. Always consistent.
                      Slower writes. Good for read-heavy, update-heavy data.

Write-behind:         Write to cache only. Async flush to DB later. Fast writes.
                      Risk: data loss if cache goes down before flush.

Read-through:         Cache sits in front of DB. App only talks to cache.
                      Cache handles DB reads on miss transparently.

TTL design:           Short TTL (5m) for availability data. Long TTL (24h) for hotel metadata.
Cache key:            search:{city}:{checkIn}:{checkOut}:{guests} → deterministic, evictable
Database choices
PostgreSQL / MySQL: Transactional data (bookings, payments). ACID guarantees. Foreign keys. Use when correctness matters more than scale.

MongoDB / DynamoDB: Flexible schema, horizontal scale. Hotel catalog (different hotels have different attributes). Read-heavy with predictable access patterns.

Redis: Cache, sessions, rate limiting, distributed locks, leaderboards. Sub-millisecond reads. Data fits in memory.

Elasticsearch: Full-text search, faceted filtering. Hotel search by name, amenities, location radius. Not for transactional data.

Cassandra: Write-heavy, time-series, globally distributed. Booking events log, analytics. Wide column store.
Message queues
Kafka:  High throughput, durable, replay-able, ordered within partition.
        Consumer groups for parallel processing. Retention-based (not ack-based).
        Use for: event streaming, audit logs, notification fanout, analytics.

SQS:    Managed, simpler, at-least-once, no ordering guarantee (FIFO queue exists).
        Auto-scales. Use for: task queues, async processing, simple decoupling.

RabbitMQ: Message broker, routing, exchanges, dead-letter queues.
          Ack-based deletion. Use for: task distribution, priority queues.
Agoda uses Kafka for their messaging pipeline. Mentioning Kafka for notification fanout and event streaming aligns with their tech stack.
Horizontal scaling patterns
Stateless services:    No session on server. State in DB/Redis. Each request to any pod.
Load balancing:        Round-robin, least connections, consistent hashing (for cache affinity).
Database sharding:     Horizontal partitioning. Shard by userId, hotelId, or geographic region.
Read replicas:         Route all reads to replicas. Write to primary only.
CQRS:                  Command (write) and Query (read) models separated. Different DBs allowed.
Event sourcing:        Store events, not state. Reconstruct state by replaying events.
Key numbers to know
L1 cache:         0.5 ns    | L2 cache:       7 ns
RAM read:         100 ns    | SSD read:       150 µs
Network (same DC):500 µs    | HDD seek:       10 ms
Network (cross DC):150 ms   | Packet CA→NL:   150 ms

1 million req/day  → ~12 req/sec
10 million req/day → ~120 req/sec
1 billion req/day  → ~12,000 req/sec

Single DB:        ~1,000 write QPS | with replicas: ~10,000 read QPS
Redis:            ~100,000 ops/sec
Kafka:            ~1,000,000 msg/sec (cluster)

Design: hotel booking system ★ Agoda
Highest probability question ▼
Agoda's core product. Very likely to appear in either platform or system design round. Know every layer.
Requirements to state upfront
Functional: Search hotels by city/dates/guests. View room availability. Book a room. Cancel booking. View booking history.

Non-functional: 10M DAU, 100K concurrent searches, 1K bookings/sec at peak. Search latency <200ms. Booking latency <2s. No double booking (strong consistency on inventory). 99.99% availability.
Data model
Hotel      (id, name, city, country, star_rating, lat, lng, amenities)
Room       (id, hotel_id, type, max_guests, price_per_night, total_count)
Inventory  (room_id, date, available_count, booked_count)  ← key table
Booking    (id, user_id, room_id, check_in, check_out, status, total_price,
            idempotency_key, created_at)
Payment    (id, booking_id, amount, currency, status, gateway_txn_id)

-- Index design (critical to mention)
-- Hotel search: INDEX (city, star_rating) + geospatial index on (lat, lng)
-- Availability check: INDEX (room_id, date) on Inventory
-- User bookings: INDEX (user_id, created_at DESC)
The hardest part: preventing double booking
Problem: 500 users request the last room simultaneously. All see available_count = 1. All proceed to book. Result: 500 bookings for 1 room.
-- Option 1: DB-level pessimistic lock (simple, slow)
BEGIN TRANSACTION;
SELECT available_count FROM inventory
WHERE room_id = ? AND date = ? FOR UPDATE; -- row-level lock
-- check count > 0, then decrement
COMMIT;

-- Option 2: Optimistic locking (faster, handles low contention)
UPDATE inventory
SET available_count = available_count - 1, version = version + 1
WHERE room_id = ? AND date = ? AND available_count > 0 AND version = ?;
-- If 0 rows affected → someone else got it → retry or fail

-- Option 3: Redis distributed lock (best for high concurrency)
SET lock:room:123:2026-07-01 {userId} NX PX 30000
-- NX = only set if not exists → only one thread acquires lock
-- PX 30000 = auto-expire in 30s (prevents deadlock)
-- If nil returned → room taken → return "sold out"
Production answer: Redis lock as first gate (fast rejection) + optimistic locking in DB as second guard + DB unique constraint as backstop. Three layers.
High-level architecture
Client → CDN → Load Balancer → API Gateway (auth, rate limit)
                                        ↓
                          ┌─────────────┴──────────────┐
                    Search Service              Booking Service
                          ↓                          ↓
                    Elasticsearch           PostgreSQL (write)
                    Redis (cache)           Redis (distributed lock)
                    Hotel Service           Payment Service
                                                  ↓
                                          Payment Gateway
                                          Kafka → Notification Service
                                                  → Email/SMS/Push

Design: hotel search system ★ Agoda
Scale + relevance + speed ▼
The core challenge
Search must handle: text queries ("Marriott Bangkok"), filters (star rating, price, amenities), geo-radius (hotels within 5km of airport), availability check (only show hotels with rooms on those dates), ranking (relevance × review score × price × commission).
Why not just SQL? Full-text search in SQL is slow. Geo queries need specialized indexes. Faceted filtering (count of results per filter value) is expensive. Ranking across multiple signals doesn't fit relational model well.

Answer: Elasticsearch for search + Redis for availability + PostgreSQL for bookings.
Search architecture
User search query
      ↓
Search Service
      ↓ (1) query Elasticsearch
Elasticsearch index (hotel metadata, amenities, location, review scores)
      ↓ returns hotel IDs with scores
      ↓ (2) filter by availability
Redis (availability bitmap per room per date)
      ↓ returns available hotel IDs
      ↓ (3) merge, rank, paginate
      ↓ (4) cache result
Redis search cache (key: city:dates:filters, TTL: 5 min)
      ↓
Response to user

Elasticsearch index design:
{
  hotel_id, name, city, country,
  location: { lat, lng },        ← geo_point type
  star_rating, review_score, price_min,
  amenities: ["wifi","pool","gym"],  ← keyword array, filterable
  tags: ["beachfront","city-center"] ← for text search
}
Keeping search index fresh
Hotel updates → Kafka topic "hotel.updated"
                     ↓
              Indexer Service (consumer)
                     ↓
              Elasticsearch update (partial)

Availability changes → do NOT index in ES (too frequent)
                    → availability stored in Redis bitmaps
                    → post-search filter step (fast O(1) per hotel)

Design: payment / wallet system ★ Agoda
Correctness is everything ▼
Core requirements
Agoda processes payments in 70+ countries, 40+ payment methods. Key constraints: no double charges, no lost payments, full audit trail, cross-currency support, PCI compliance.
Data model
Wallet     (id, user_id, currency, balance, version)  ← version for optimistic lock
Transaction(id, wallet_id, type, amount, currency, status,
            reference_id, idempotency_key, created_at)
           type: CREDIT | DEBIT | REFUND | FEE
           status: PENDING | COMPLETED | FAILED | REVERSED

Payment    (id, booking_id, amount, currency, method, gateway,
            gateway_txn_id, status, idempotency_key)
The double-debit problem
Scenario: User pays. Request times out. Client retries. Payment charged twice.

-- Solution: idempotency key (client-generated UUID per payment attempt)
-- Server checks: has this idempotency_key been processed?
-- Yes → return cached result (same response as original)
-- No → process payment, store result against key

-- Also: outbox pattern for exactly-once event publishing
BEGIN TRANSACTION;
  UPDATE wallet SET balance = balance - amount WHERE id = ?;
  INSERT INTO outbox (event_type, payload) VALUES ('PAYMENT_PROCESSED', ?);
COMMIT;
-- Separate process reads outbox, publishes to Kafka, deletes row
-- Guarantees: payment and event published atomically
Refund flow
Refund is NOT a reversal — it's a new credit transaction.
This maintains a complete audit trail.

Refund request → validate (booking cancelled? refund policy?)
              → create REFUND transaction (PENDING)
              → call payment gateway refund API
              → on success: update to COMPLETED, credit wallet
              → on failure: mark FAILED, alert, manual review queue
              → publish RefundCompleted event → email notification

Design: notification system ★ Agoda
Async, multi-channel, at-scale ▼
Key insight: notifications are async
Never make the booking API wait for email/SMS to send. A slow email provider would make every booking slow. Decouple using Kafka.
Booking Service → Kafka topic "booking.events"
                          ↓
                 Notification Service (consumer)
                 reads event type → routes to channel
                          ↓
              ┌───────────┼──────────────┐
         Email Service  SMS Service  Push Service
         (SendGrid)     (Twilio)     (FCM/APNs)
              ↓               ↓            ↓
         User email      User phone   User app
Reliability: what if email service is down?
Kafka retains messages. Consumer fails → messages stay in partition.
When email service recovers → picks up from last committed offset.

Dead letter queue (DLQ): after 3 retries with exponential backoff,
move to DLQ topic. Alert on-call. Manual review.

Exponential backoff:
  Retry 1: wait 10s
  Retry 2: wait 30s
  Retry 3: wait 90s
  Then → DLQ
User preferences
NotificationPreference (user_id, channel, event_type, enabled)
-- Before sending: check preferences
-- User opted out of SMS for booking confirmations? Skip SMS.
-- User in quiet hours (23:00-07:00)? Defer push, not email.

Design: rate limiter generic
Very commonly asked ▼
Algorithms — know all four
Token Bucket:   Bucket holds max N tokens. Refill at rate R/sec.
                Each request consumes 1 token. If empty → reject 429.
                Allows burst up to bucket size. Most common.

Leaking Bucket: Queue of requests processed at fixed rate.
                Smooth output. Excess dropped. Good for APIs with
                strict throughput guarantees.

Fixed Window:   Count requests in current time window (1 min).
                Problem: burst at window boundary — 2× allowed in 1 second.

Sliding Window: More accurate. Weight recent counts.
                More complex. Redis ZSET with timestamps.
Distributed rate limiter with Redis
-- Token bucket in Redis (atomic with Lua script)
local tokens = redis.call('GET', key)
if tokens == nil then tokens = MAX_TOKENS end
if tokens > 0 then
  redis.call('DECR', key)
  redis.call('EXPIRE', key, WINDOW_SECONDS)
  return 1  -- allowed
else
  return 0  -- rejected 429
end

-- Sliding window with sorted set
ZADD user:123:requests {timestamp} {uuid}
ZREMRANGEBYSCORE user:123:requests 0 {timestamp - windowMs}
count = ZCARD user:123:requests
if count > limit then reject else allow

-- Rate limit keys: by userId, by IP, by API key, by endpoint
-- Headers to return: X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After

Generic designs — cover the basics
URL shortener · Feed system · Chat ▼
URL shortener (TinyURL)
Key decisions:
  Short code: 6 chars from [a-zA-Z0-9] = 62^6 = 56 billion combinations
  Generation: Base62 encode auto-increment ID (avoids collision entirely)
  Storage: (short_code → original_url, user_id, created_at, expires_at)
  Redirect: HTTP 301 (permanent, browser caches) vs 302 (temporary, we track)
            Use 302 to track click analytics
  Scale: Read-heavy (1000:1). Cache top short URLs in Redis (LRU).
News feed / activity feed
Fan-out on write (push):  When user posts, immediately push to all followers' feeds.
                          Fast reads. Slow writes. Bad for celebrities (10M followers).

Fan-out on read (pull):   On feed load, fetch posts from all followees. Merge sort by time.
                          Slow reads. Good for celebrities.

Hybrid:                   Push for regular users. Pull for celebrity accounts (>1M followers).
                          Agoda-equivalent: push hotel deals to subscribed users.

Feed table: (user_id, post_id, created_at) ← one row per user per feed item
Served from Redis sorted set: ZADD feed:{userId} {timestamp} {postId}
Chat system (WhatsApp-style)
WebSocket for real-time: persistent connection, server pushes messages.
  HTTP polling: inefficient, creates many connections.
  Long polling: better, but still many connections.
  WebSocket: best, bidirectional, one connection per user.

Message delivery guarantee:
  1. Client sends message → server ACKs receipt (single tick)
  2. Server stores in DB, delivers to recipient
  3. Recipient device ACKs receipt (double tick)
  4. Recipient reads → read receipt (blue ticks)

Storage: messages in Cassandra (write-heavy, time-series friendly)
Presence (online/offline): Redis with TTL on user:online:{userId}
