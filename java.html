OOP — Four Pillars
The foundation of every Java interview
Beginner▼
Why it matters
Concept
Code
Real-world
Why interviewers always start here
OOP principles underpin every design pattern, every framework, and every architecture decision. When an interviewer asks "how would you design a payment system?" — your answer reveals whether you actually understand encapsulation and abstraction, or just memorised definitions.
Encapsulation — hide internal state, expose only what's needed. Your WFM Forecasting service doesn't expose its internal calculation state to other services. It exposes a clean API.

Inheritance — reuse behaviour via parent classes. A BookingException extends RuntimeException — it gets all exception behaviour for free.

Polymorphism — same interface, different implementations. PaymentStrategy has multiple implementations (wallet, card, UPI) — the caller doesn't care which one.

Abstraction — expose what, hide how. HotelRepository.findByCity() — callers don't know if it hits PostgreSQL, Redis, or Elasticsearch.
The four pillars — precise definitions
PillarWhat it meansHow violated
EncapsulationBundle data + methods. Private fields, public getters/setters only when needed.Public fields that anyone can modify directly
InheritanceChild class inherits parent's fields and methods. IS-A relationship.Deep inheritance hierarchies (prefer composition)
PolymorphismOne interface, many forms. Method overriding (runtime) and overloading (compile-time).Giant if-else / switch checking object types
AbstractionHide implementation details behind interfaces/abstract classes. Expose only behaviour.Leaking implementation details into API signatures
Composition over Inheritance — the most important principle in modern Java. Instead of class Dog extends Animal, prefer class Dog { private AnimalBehaviour behaviour; }. Composition is more flexible and avoids the fragile base class problem.
// Encapsulation — private state, controlled access
public class BankAccount {
    private double balance; // nobody can touch this directly

    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Amount must be positive");
        balance += amount; // validation happens here, not in caller
    }
    public double getBalance() { return balance; } // read-only view
}

// Polymorphism — same method call, different behaviour at runtime
interface Notification { void send(String message); }

class EmailNotification implements Notification {
    public void send(String msg) { System.out.println("Email: " + msg); }
}
class SMSNotification implements Notification {
    public void send(String msg) { System.out.println("SMS: " + msg); }
}

// Caller doesn't know or care which implementation it gets
void notifyUser(Notification n) { n.send("Booking confirmed"); }
// notifyUser(new EmailNotification()) → "Email: Booking confirmed"
// notifyUser(new SMSNotification())  → "SMS: Booking confirmed"

// Abstraction — abstract class defines template, children fill in details
abstract class ReportGenerator {
    public final void generate() { // Template Method pattern
        fetchData();
        processData();
        exportReport(); // each subclass implements differently
    }
    protected abstract void fetchData();
    protected abstract void processData();
    protected abstract void exportReport();
}

Strings — Immutability & StringBuilder
Why string concatenation in a loop is a bug
Beginner▼
Concept
Code
Immutability
In Java, String objects are immutable — once created, they cannot be changed. When you do s = s + "world", Java doesn't modify the original string. It creates a brand new String object and discards the old one. In a loop of 1000 iterations, that's 1000 objects created and garbage collected — O(n²) total work.

String (immutable)

Thread-safe. Every operation creates new object. Use for short or static strings. String literals go in String Pool (memory efficient).

StringBuilder (mutable)

Not thread-safe. Operations modify same object. Use in single-threaded code for building strings dynamically. O(n) total instead of O(n²).

StringBuffer is like StringBuilder but synchronized — thread-safe but slow. Almost never needed; prefer StringBuilder + explicit synchronization when needed.
// BAD — O(n²) — creates 1000 String objects
String result = "";
for (int i=0; i<1000; i++) result += i; // new String every iteration

// GOOD — O(n) — one StringBuilder object
StringBuilder sb = new StringBuilder();
for (int i=0; i<1000; i++) sb.append(i);
String result = sb.toString();

// String API you need cold
String s = "Hello World";
s.length()           // 11
s.charAt(0)          // 'H'
s.substring(0,5)     // "Hello"
s.indexOf("World")   // 6
s.toLowerCase()      // "hello world"
s.trim()             // remove leading/trailing whitespace
s.split(" ")         // ["Hello", "World"]
s.toCharArray()      // ['H','e','l','l','o',' ','W','o','r','l','d']
s.contains("World") // true
s.replace("World","Java") // "Hello Java"
String.join("-", "a", "b", "c") // "a-b-c"

// Comparing strings — NEVER use == for content comparison
"hello".equals("hello")         // true — correct
"hello" == "hello"              // true ONLY for string literals (pool)
new String("hello") == "hello" // false — different objects!

// StringBuilder API
StringBuilder sb = new StringBuilder("hello");
sb.append(" world");   // "hello world"
sb.insert(5, ",");      // "hello, world"
sb.reverse();           // "dlrow ,olleh"

Exception Handling
Checked vs unchecked, try-with-resources, custom exceptions
Beginner▼
Concept
Code
Exception hierarchy
Throwable ├── Error (don't catch — OutOfMemoryError, StackOverflowError) └── Exception ├── Checked Exception (must declare or catch — IOException, SQLException) └── RuntimeException (unchecked — NullPointerException, IllegalArgumentException) ↑ Your custom business exceptions should extend this

Checked Exceptions

Must be declared in method signature (throws IOException) or caught. Caller is forced to handle it. Use for recoverable external conditions: file not found, network failure.

Unchecked (Runtime)

Not required to declare or catch. Use for programming errors: null dereference, invalid arguments, out of bounds. Your custom business exceptions (BookingException) should be unchecked.

// Custom business exception — extends RuntimeException (unchecked)
public class BookingException extends RuntimeException {
    private final String errorCode;

    public BookingException(String message, String errorCode) {
        super(message);
        this.errorCode = errorCode;
    }
    public BookingException(String message, String errorCode, Throwable cause) {
        super(message, cause); // preserve original stack trace
        this.errorCode = errorCode;
    }
    public String getErrorCode() { return errorCode; }
}

// try-with-resources — ALWAYS use for resources (auto-closes)
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql)) {
    // conn and ps auto-closed even if exception thrown
    ps.executeQuery();
} catch (SQLException e) {
    // handle specific exception
    log.error("DB error: {}", e.getMessage(), e);
    throw new BookingException("Failed to load booking", "DB_ERROR", e);
}

// Multi-catch (Java 7+)
try { riskyOperation(); }
catch (IOException | TimeoutException e) {
    // handle both with same logic
}

// NEVER do this — swallows exception silently
try { riskyOperation(); } catch (Exception e) { } // BAD!

// NEVER do this — too broad, hides bugs
try { riskyOperation(); } catch (Exception e) { e.printStackTrace(); } // BAD!

// Filter, map, collect — most common pipeline
List<String> premiumHotelNames = hotels.stream()
    .filter(h -> h.getStarRating() >= 4)
    .filter(h -> h.isAvailable())
    .sorted((a,b) -> b.getReviewScore().compareTo(a.getReviewScore()))
    .limit(10)
    .map(Hotel::getName)
    .collect(toList());

// groupingBy — group bookings by hotel
Map<String, List<Booking>> byHotel = bookings.stream()
    .collect(groupingBy(Booking::getHotelId));

// groupingBy with downstream collector — count per hotel
Map<String, Long> bookingCount = bookings.stream()
    .collect(groupingBy(Booking::getHotelId, counting()));

// toMap — convert to Map
Map<Long, Hotel> hotelById = hotels.stream()
    .collect(toMap(Hotel::getId, h -> h));

// reduce — sum prices
double totalRevenue = bookings.stream()
    .mapToDouble(Booking::getPrice)
    .sum(); // or .reduce(0.0, Double::sum)

// flatMap — flatten nested lists
List<Room> allRooms = hotels.stream()
    .flatMap(h -> h.getRooms().stream())
    .collect(toList());

// Optional — avoid null checks
Optional<Hotel> found = hotels.stream()
    .filter(h -> h.getId().equals(targetId))
    .findFirst();
found.ifPresent(h -> log.info("Found: {}", h.getName()));
String name = found.map(Hotel::getName).orElse("Unknown");

// Method references — cleaner than lambdas
hotels.stream().map(Hotel::getName)     // instance method ref
hotels.stream().forEach(System.out::println) // static method ref
hotels.stream().map(Hotel::new)         // constructor ref

Generics
Type-safe code without casting — wildcards, bounds, type erasure
Intermediate▼
Concept
Code
Why generics exist
Before generics, collections stored Object. You could add a String and an Integer to the same list — the bug only appeared at runtime when you tried to cast. Generics move these errors to compile time.

<T extends Foo>

Upper bound. T must be Foo or a subclass. You can READ from it — T is at least a Foo. Use for producers: methods that give you T values.

<T super Foo>

Lower bound. T must be Foo or a superclass. You can WRITE to it. Use for consumers: methods that accept T values.

PECS rule: Producer Extends, Consumer Super. If you're reading (producing) from a collection → use ? extends T. If you're writing (consuming) to a collection → use ? super T.
// Generic class
public class ApiResponse<T> {
    private T data;
    private String status;
    private String message;

    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>(data, "SUCCESS", null);
    }
    public static <T> ApiResponse<T> error(String message) {
        return new ApiResponse<>(null, "ERROR", message);
    }
}
// Usage: ApiResponse<List<Hotel>> response = ApiResponse.success(hotels);

// Generic method
public static <T extends Comparable<T>> T max(List<T> list) {
    return list.stream().max(Comparator.naturalOrder()).orElseThrow();
}

// Wildcards — PECS rule
// Producer (reading from list) — ? extends
public double totalPrice(List<? extends Booking> bookings) {
    return bookings.stream().mapToDouble(Booking::getPrice).sum();
}
// Consumer (writing to list) — ? super
public void addBookings(List<? super Booking> dest, List<Booking> src) {
    dest.addAll(src);
}

The threading problem
A race condition occurs when two or more threads access shared state concurrently and at least one modifies it, without proper synchronization. The result is unpredictable — depends on thread scheduling, which varies every run.

// volatile — visibility only, NOT atomicity
private volatile boolean isRunning = true;
// Thread A: isRunning = false → Thread B sees it immediately
// But: isRunning++ is NOT safe with volatile (3 operations: read, inc, write)

// AtomicInteger — lock-free atomic operations via CAS
private AtomicInteger bookingCount = new AtomicInteger(0);
bookingCount.incrementAndGet();             // atomic ++
bookingCount.addAndGet(5);                 // atomic += 5
bookingCount.compareAndSet(10, 0);        // set to 0 only if currently 10

// synchronized — one thread at a time
private final Object lock = new Object();
public void transferFunds(double amount) {
    synchronized (lock) { // block-level lock
        balance -= amount;
        processedCount++;
    }
}

// ReentrantLock — explicit lock with timeout
private final ReentrantLock lock = new ReentrantLock();
public boolean tryTransfer(double amount) {
    if (lock.tryLock(100, TimeUnit.MILLISECONDS)) {
        try { balance -= amount; return true; }
        finally { lock.unlock(); } // ALWAYS unlock in finally
    }
    return false; // couldn't acquire lock in time
}

// ConcurrentHashMap — thread-safe without locking whole map
Map<String, Integer> cache = new ConcurrentHashMap<>();
cache.putIfAbsent("key", 0);     // atomic check-then-put
cache.computeIfAbsent("key", k -> expensiveLoad(k)); // compute once
cache.merge("key", 1, Integer::sum); // atomic increment
TRex multi-tenant onboarding: When onboarding multiple tenants concurrently to Staging/Prod-US/Prod-EU, a ConcurrentHashMap tracks onboarding state per tenant. AtomicInteger counts successfully onboarded tenants. If one tenant fails, others continue — no shared mutable state between tenant onboarding tasks.

WFM Forecasting: The forecasting service processes multiple tenants in parallel using a thread pool (ExecutorService). Each tenant's forecast is independent (no shared state), so no synchronization needed. Results are collected using CompletableFuture.allOf().

People Fabric CDC: The pipeline processes CDC events from PostgreSQL in order per entity but can process different entities in parallel. BlockingQueue separates the reader thread (reads CDC events) from writer threads (insert to AlloyDB).

CompletableFuture — Async Java
Non-blocking async operations — the modern way
Intermediate▼
Concept
Code
Why async matters for booking systems
A hotel search needs: hotel data + room availability + current pricing + user's past bookings. If done sequentially: 4 × 50ms = 200ms. If done in parallel: ~50ms (limited by slowest). CompletableFuture enables this without blocking threads.
Sequential: [hotel] → [availability] → [pricing] → [history] = 200ms Async parallel: [hotel ] [availability] ← all start at same time [pricing ] [history ] = ~50ms (slowest task)
// Run multiple async operations and combine
CompletableFuture<Hotel> hotelFuture =
    CompletableFuture.supplyAsync(() -> hotelService.findById(hotelId));

CompletableFuture<List<Room>> roomsFuture =
    CompletableFuture.supplyAsync(() -> roomService.findAvailable(hotelId, dates));

CompletableFuture<Double> priceFuture =
    CompletableFuture.supplyAsync(() -> pricingService.getCurrentPrice(hotelId));

// Wait for all, then combine
CompletableFuture<HotelSearchResult> result =
    CompletableFuture.allOf(hotelFuture, roomsFuture, priceFuture)
        .thenApply(v -> new HotelSearchResult(
            hotelFuture.join(),
            roomsFuture.join(),
            priceFuture.join()
        ));

// Chain operations
CompletableFuture<String> pipeline =
    CompletableFuture.supplyAsync(() -> fetchUserData(userId))    // async
        .thenApply(user -> user.getPremiumStatus())               // transform (still async)
        .thenApplyAsync(status -> loadPremiumDeals(status))       // on different thread
        .exceptionally(ex -> "standard-deals");                   // fallback on error

// With timeout (Java 9+)
CompletableFuture<Hotel> withTimeout =
    hotelFuture.orTimeout(2, TimeUnit.SECONDS);

// Race — take whichever resolves first
CompletableFuture<String> fastest =
    CompletableFuture.anyOf(primaryCache, secondaryCache, dbFallback)
        .thenApply(result -> (String) result);

JVM memory areas
JVM Memory ├── Heap (GC-managed) │ ├── Young Generation │ │ ├── Eden Space ← new objects allocated here │ │ └── Survivor (S0,S1) ← survived objects move here │ └── Old Generation ← long-lived objects promoted here ├── Metaspace (off-heap) ← class metadata (replaced PermGen in Java 8) ├── Stack (per thread) ← local variables, method call frames ├── Native Memory ← NIO buffers, JIT compiled code └── Code Cache ← JIT compiled native code
Young vs Old gen: Most objects die young (short-lived request-scoped objects). Minor GC collects Young Gen — fast (milliseconds). Major/Full GC collects Old Gen — slow (can cause stop-the-world pauses of hundreds of ms). Tune heap ratio to minimize Full GC.

// Singleton — thread-safe with double-checked locking
public class ConfigService {
    private volatile static ConfigService instance;

    public static ConfigService getInstance() {
        if (instance == null) {
            synchronized (ConfigService.class) {
                if (instance == null) instance = new ConfigService();
            }
        }
        return instance;
    }
}

// Builder — for complex objects with many optional fields
public class Booking {
    private final String userId;     private final String roomId;
    private final LocalDate checkIn; private final LocalDate checkOut;
    private final String couponCode; // optional
    private final String notes;       // optional

    private Booking(Builder b) { /* assign from builder */ }

    public static class Builder {
        private String userId, roomId, couponCode, notes;
        private LocalDate checkIn, checkOut;

        public Builder userId(String v) { userId=v; return this; }
        public Builder roomId(String v) { roomId=v; return this; }
        public Builder checkIn(LocalDate v) { checkIn=v; return this; }
        public Builder coupon(String v) { couponCode=v; return this; }

// Strategy — swap algorithm at runtime
interface PricingStrategy { double calculate(double basePrice, BookingContext ctx); }

class SeasonalPricing implements PricingStrategy {
    public double calculate(double base, BookingContext ctx) {
        return ctx.isPeakSeason() ? base * 1.3 : base;
    }
}

// Observer — react to events without coupling
interface BookingEventListener { void onBookingCreated(BookingEvent event); }

class BookingEventPublisher {
    private List<BookingEventListener> listeners = new ArrayList<>();
    public void subscribe(BookingEventListener l) { listeners.add(l); }
    public void publish(BookingEvent event) {
        listeners.parallelStream().forEach(l -> l.onBookingCreated(event));
    }
}

// Template Method — define skeleton, let subclasses fill details
abstract class BookingProcessor {
    public final BookingResult process(BookingRequest req) {
        validateRequest(req);
        Inventory inv = checkInventory(req);
        Payment payment = processPayment(req);
        return createBookingRecord(req, inv, payment);
    }
    protected abstract void validateRequest(BookingRequest req);
    protected abstract Inventory checkInventory(BookingRequest req);
    protected abstract Payment processPayment(BookingRequest req);
    protected abstract BookingResult createBookingRecord(BookingRequest req, Inventory inv, Payment p);
}

for (String col : columns) sql += col + ","; // O(n²)
// GOOD:
String sql = String.join(",", columns); // O(n)

// 2. Autoboxing in hot paths — creates Integer objects for each int
// BAD (in a tight loop):
Integer sum = 0;
for (int i : largeList) sum += i; // autoboxes int to Integer each time
// GOOD:
int sum = 0;
for (int i : largeList) sum += i;

// 3. Eagerly loading large collections — load only what you need
// BAD:
List<Hotel> all = hotelRepo.findAll(); // loads ALL hotels into memory
// GOOD:
Page<Hotel> page = hotelRepo.findAll(PageRequest.of(0, 20));

// 4. N+1 queries (ORM anti-pattern)
// BAD: fetching rooms for each hotel in a loop
for (Hotel h : hotels) h.getRooms().size(); // N queries!
// GOOD: @EntityGraph or JOIN FETCH or @BatchSize
@Query("SELECT h FROM Hotel h JOIN FETCH h.rooms WHERE h.city = :city")
List<Hotel> findByCityWithRooms(String city);

// 5. Using Stream.collect() when a simple loop suffices
// For side effects, forEach is fine. Collect is for producing new collections.
hotels.forEach(h -> cache.put(h.getId(), h)); // cleaner
