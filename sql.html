Optimistic vs Pessimistic Locking — Preventing Double Booking
The most important Staff-level SQL question for booking systems
Intermediate▼
Pessimistic (SELECT FOR UPDATE)
Optimistic (@Version)
When to use which
Pessimistic Locking — Lock the row at read time
Idea: When you read the inventory row, acquire an exclusive lock immediately. No other transaction can read or write that row until you commit or rollback. Best for high-contention scenarios — last room, flash sales, limited seats.
-- Two users hit Book simultaneously for room 101 on 2026-07-09
-- User A and User B both run this transaction

BEGIN TRANSACTION;

-- SELECT FOR UPDATE: locks this row exclusively
-- User B blocks here until User A commits or rolls back
SELECT available_count
FROM room_inventory
WHERE room_id = 101 AND date = '2026-07-09'
FOR UPDATE;

-- User A gets available_count = 1 (last room)
-- User A proceeds:
UPDATE room_inventory
SET available_count = available_count - 1
WHERE room_id = 101 AND date = '2026-07-09';

INSERT INTO bookings (room_id, user_id, date, status)
VALUES (101, 'user_a', '2026-07-09', 'CONFIRMED');

COMMIT;
-- Lock released. User B unblocks.
-- User B's SELECT FOR UPDATE now returns available_count = 0
-- User B sees no availability → return error to user
// Spring Boot / JPA implementation
@Repository
public interface RoomInventoryRepository extends JpaRepository<RoomInventory, Long> {

  // @Lock(PESSIMISTIC_WRITE) = SELECT ... FOR UPDATE in SQL
  @Lock(LockModeType.PESSIMISTIC_WRITE)
  @Query("SELECT r FROM RoomInventory r WHERE r.roomId = :roomId AND r.date = :date")
  Optional<RoomInventory> findForUpdate(@Param("roomId") Long roomId,
                                         @Param("date") LocalDate date);
}

@Service
public class BookingService {

  @Transactional  // CRITICAL: must be inside a transaction
  public Booking createBooking(Long roomId, LocalDate date, Long userId) {
    RoomInventory inv = inventoryRepo
      .findForUpdate(roomId, date)              // acquires lock
      .orElseThrow(() -> new NotFoundException());

    if (inv.getAvailableCount() <= 0) {
      throw new RoomNotAvailableException("No rooms left");
    }

    inv.setAvailableCount(inv.getAvailableCount() - 1);
    inventoryRepo.save(inv);                    // UPDATE

    Booking booking = new Booking(roomId, userId, date, BookingStatus.CONFIRMED);
    return bookingRepo.save(booking);           // INSERT
    // Transaction commits here → lock released
  }
}
Risk of pessimistic locking: Deadlocks. If Transaction A locks room 101 then tries to lock room 102, while Transaction B locked room 102 and tries to lock room 101 — deadlock. Fix: always acquire locks in a consistent order (by room_id ASC). MySQL detects deadlocks and auto-rolls back one transaction.
Optimistic Locking — Check at write time, not read time
Idea: Don't lock at all. Add a version column. When you update, verify the version hasn't changed since you read it. If it has, someone else modified the row — throw an exception and let the caller retry. Best for low-contention scenarios.
-- Schema: version column tracks row revision
CREATE TABLE room_inventory (
  room_id    BIGINT,
  date       DATE,
  available  INT,
  version    INT DEFAULT 0,   -- incremented on every UPDATE
  PRIMARY KEY (room_id, date)
);

-- User A reads: available=1, version=5
-- User B reads: available=1, version=5 (same read, no lock)

-- User A writes first:
-- WHERE version=5 ensures no one else modified since we read
UPDATE room_inventory
SET available = available - 1, version = version + 1
WHERE room_id = 101 AND date = '2026-07-09' AND version = 5;
-- Rows affected: 1 → success, now version=6

-- User B writes second (version=5 no longer exists):
-- WHERE version=5 matches nothing → 0 rows affected
UPDATE room_inventory
SET available = available - 1, version = version + 1
WHERE room_id = 101 AND date = '2026-07-09' AND version = 5;
-- Rows affected: 0 → OptimisticLockException → retry or error
// JPA: @Version does this automatically
@Entity
public class RoomInventory {
  @Id
  private Long roomId;

  @Version                   // JPA auto-manages this column
  private Integer version;   // included in WHERE clause of every UPDATE

  private Integer available;
}

// If version mismatch on save() → throws OptimisticLockException
// Caller catches it and retries, or returns 409 Conflict to client
try {
  bookingService.createBooking(roomId, date, userId);
} catch (OptimisticLockException e) {
  throw new ResponseStatusException(HttpStatus.CONFLICT,
    "Room just got booked by someone else, please try again");
}

💡 Interview answer template
"For the final booking write, we use pessimistic locking via SELECT FOR UPDATE — because at the moment of booking, contention is high and the window is small, so the cost of blocking is acceptable. For inventory availability checks (which happen on every search), we don't lock at all — we read the current count and accept slight staleness. The locking only kicks in when the user actually commits to booking."

Transaction Isolation Levels — What Each One Allows
Dirty read · Non-repeatable read · Phantom read — what these mean and when they bite you
Intermediate▼ 
-- The 4 isolation levels (weakest → strongest)
-- MySQL default: REPEATABLE READ
-- PostgreSQL default: READ COMMITTED

-- READ UNCOMMITTED (never use in production)
-- Allows dirty reads: you can read data another transaction hasn't committed yet
-- If that transaction rolls back, you read data that never existed
-- Problem: User A updates price to $50 but rolls back. User B read $50 during that time.

-- READ COMMITTED (PostgreSQL default)
-- No dirty reads. But allows non-repeatable reads:
-- Same SELECT within one transaction can return different results if another tx commits between reads
BEGIN; -- Transaction A
SELECT available FROM room_inventory WHERE room_id = 101; -- returns 2
-- Transaction B commits: UPDATE room_inventory SET available = 1 WHERE room_id = 101
SELECT available FROM room_inventory WHERE room_id = 101; -- returns 1 (changed!)
COMMIT;

-- REPEATABLE READ (MySQL InnoDB default)
-- Same SELECT always returns same rows within a transaction
-- But allows phantom reads: new rows inserted by other transactions ARE visible
BEGIN; -- Transaction A
SELECT COUNT(*) FROM bookings WHERE date = '2026-07-09'; -- returns 10
-- Transaction B commits: INSERT INTO bookings ... (adds new row for 2026-07-09)
SELECT COUNT(*) FROM bookings WHERE date = '2026-07-09'; -- still 10 (snapshot isolation)
-- MySQL's MVCC prevents phantom reads in REPEATABLE READ — PostgreSQL doesn't

-- SERIALIZABLE (strongest, slowest)
-- Transactions execute as if sequential. No dirty, non-repeatable, or phantom reads.
-- Use for: financial summaries, inventory audits where exact counts matter
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN;
-- Full table-range locks prevent any insert/update that affects our query range

SQL Indexing — B-Tree, Composite, Covering Indexes
How MySQL uses indexes and why the left-prefix rule matters for your queries
Beginner▼ 
-- B-Tree index: data sorted in a balanced tree
-- Range queries, ORDER BY, equality — all benefit from B-tree
-- MySQL InnoDB: clustered B-tree on PRIMARY KEY; secondary indexes hold primary key values

-- Composite index: covers multiple columns
-- Stored sorted by (col1, then col2, then col3)
CREATE INDEX idx_booking_search
ON bookings (user_id, status, check_in);

-- Left-prefix rule: index is usable only if query starts from the LEFT
-- This index helps these queries:
SELECT * FROM bookings WHERE user_id = 'u1';                             -- ✓ uses index
SELECT * FROM bookings WHERE user_id = 'u1' AND status = 'CONFIRMED';    -- ✓ uses both cols
SELECT * FROM bookings WHERE user_id = 'u1' AND status = 'CONFIRMED'
                        AND check_in > '2026-07-01';                     -- ✓ uses all 3

-- These queries CANNOT use the index efficiently:
SELECT * FROM bookings WHERE status = 'CONFIRMED';                       -- ✗ skips user_id (leftmost)
SELECT * FROM bookings WHERE check_in > '2026-07-01';                    -- ✗ skips first two cols

-- Covering index: index contains ALL columns the query needs
-- Avoids hitting the actual table row (faster)
CREATE INDEX idx_room_availability
ON room_inventory (room_id, date, available_count);  -- available_count in index

-- This query is served entirely from the index — no table lookup needed:
SELECT available_count FROM room_inventory
WHERE room_id = 101 AND date = '2026-07-09';

-- EXPLAIN: check if your query uses an index
EXPLAIN SELECT * FROM bookings WHERE user_id = 'u1' AND status = 'CONFIRMED';
-- key: idx_booking_search  ← good, using index
-- type: ref               ← good (eq_ref, ref, range are all good)
-- type: ALL               ← BAD: full table scan — add an index
-- rows: 1                 ← estimated rows examined (lower = better)
-- Extra: Using index      ← covering index being used (best case)
InnoDB nuance: Secondary indexes in InnoDB store the primary key, not the row pointer. So a query that hits a secondary index then needs a primary key lookup if it needs columns not in the index. This is why covering indexes (all needed columns are in the index) are faster — they avoid the second lookup.

N+1 Query Problem — The JPA Performance Trap
How to detect it with Hibernate logging and fix it with JOIN FETCH or @EntityGraph
Intermediate▼ 
// PROBLEM: Get 100 hotels with their rooms
// This looks fine but runs 101 SQL queries

@Entity
public class Hotel {
  @Id private Long id;
  private String name;

  @OneToMany(mappedBy = "hotel", fetch = FetchType.LAZY)  // LAZY = loaded on demand
  private List<Room> rooms;
}

// Service code that causes N+1:
List<Hotel> hotels = hotelRepo.findAll();              // Query 1: SELECT * FROM hotels (100 rows)
for (Hotel h : hotels) {
  h.getRooms().size();                                 // Query 2-101: SELECT * FROM rooms WHERE hotel_id=?
                                                       // One query per hotel = 100 extra queries
}
// Total: 1 + N = 101 queries. For 1000 hotels: 1001 queries. Production killer.

// DETECT with Hibernate SQL logging in application.properties:
// spring.jpa.show-sql=true
// logging.level.org.hibernate.stat=DEBUG
// spring.jpa.properties.hibernate.generate_statistics=true
// Look for: HQL queries fired: 101

// FIX 1: JOIN FETCH (one SQL query with JOIN)
@Query("SELECT h FROM Hotel h JOIN FETCH h.rooms WHERE h.cityId = :cityId")
List<Hotel> findWithRoomsByCityId(@Param("cityId") Long cityId);
// SQL: SELECT h.*, r.* FROM hotels h JOIN rooms r ON r.hotel_id = h.id WHERE h.city_id = ?
// Result: 1 query instead of N+1

// FIX 2: @EntityGraph (declarative approach)
@EntityGraph(attributePaths = {"rooms", "amenities"})
@Query("SELECT h FROM Hotel h WHERE h.cityId = :cityId")
List<Hotel> findWithDetails(@Param("cityId") Long cityId);
// Fetches rooms AND amenities in one LEFT OUTER JOIN query

// FIX 3: @BatchSize (compromise — still N queries but batched)
// @BatchSize(size=50) on the collection — fetches 50 hotels' rooms in one IN clause
@OneToMany(fetch = FetchType.LAZY)
@BatchSize(size = 50)
private List<Room> rooms;
JOIN FETCH caveat: JOIN FETCH with pagination (setMaxResults) causes Hibernate to load ALL rows into memory and paginate in Java, not SQL. This is a silent performance killer. When you need pagination + eager loading, use two separate queries: one for the page of IDs, one JOIN FETCH for those specific IDs.

EXPLAIN & Slow Query Debugging
How to read a query plan and fix slow queries in production
Advanced▼ 
-- EXPLAIN: what MySQL does with your query
EXPLAIN SELECT b.id, h.name, b.check_in, b.total_price
FROM bookings b
JOIN hotels h ON b.hotel_id = h.id
WHERE b.user_id = 'u123'
AND b.status = 'CONFIRMED'
AND b.check_in > '2026-07-01'
ORDER BY b.check_in DESC
LIMIT 20;

-- Example output (read left-to-right, inner first):
-- id | table | type   | possible_keys              | key              | rows | Extra
-- 1  | b     | ref    | idx_user_status,idx_user   | idx_user_status  | 15   | Using where; Using filesort
-- 1  | h     | eq_ref | PRIMARY                    | PRIMARY          | 1    |

-- WHAT TO LOOK FOR:
-- type column (best to worst):
--   const    → single row by primary key or unique key (best)
--   eq_ref   → single row lookup per outer row (JOIN on unique key)
--   ref      → multiple rows matching non-unique index
--   range    → index range scan (WHERE col BETWEEN / > / <)
--   index    → full index scan (better than ALL but still slow for large tables)
--   ALL      → FULL TABLE SCAN — almost always a problem on large tables

-- Extra column warning signs:
--   Using filesort  → no index covers the ORDER BY → slow for large result sets
--   Using temporary → temp table created (GROUP BY without index)
--   Using where     → filtering after index access (usually fine)

-- FIX "Using filesort" for ORDER BY check_in DESC:
-- Add check_in to the index so ORDER is already sorted in the index
CREATE INDEX idx_booking_user_status_checkin
ON bookings (user_id, status, check_in DESC);
-- Now ORDER BY check_in DESC is free (index already ordered that way)

-- EXPLAIN ANALYZE (MySQL 8.0+): actual vs estimated rows
EXPLAIN ANALYZE SELECT ...;
-- Shows: actual time=0.5..2.3 rows=15 loops=1
-- If actual rows >> estimated rows → statistics stale → run ANALYZE TABLE bookings

-- Slow query log: find real production slow queries
-- my.cnf:
-- slow_query_log = 1
-- slow_query_log_file = /var/log/mysql/slow.log
-- long_query_time = 1.0   ← log queries taking > 1 second
-- log_queries_not_using_indexes = 1

Connection Pooling — HikariCP in Spring Boot
Why you need it, how to size it, what goes wrong when it's misconfigured
Advanced▼ 
# application.yml — HikariCP (Spring Boot default pool)
spring:
  datasource:
    url: jdbc:mysql://prod-db:3306/agoda_bookings
    username: "${DB_USER}"
    password: "${DB_PASSWORD}"
    hikari:
      maximum-pool-size: 20      # max DB connections (NOT more than DB can handle)
      minimum-idle: 5            # keep 5 connections warm at idle
      connection-timeout: 30000  # wait max 30s for a connection from pool
      idle-timeout: 600000       # close idle connections after 10min
      max-lifetime: 1800000      # recycle connections every 30min (avoids stale connections)
      leak-detection-threshold: 5000  # warn if connection held > 5s (leaks!)

# Pool sizing rule: don't guess, measure
# MySQL max_connections default: 151
# If you have 10 Spring Boot pods × 20 pool = 200 connections
# → Exceeds MySQL limit → connection refused errors
# 
# Formula (Hikari docs): connections = (core_count × 2) + effective_spindle_count
# For a 4-core DB server: ~9 connections per app instance
#
# Key failure modes:
# 1. Pool exhaustion: all 20 connections in use, new request waits → timeout after 30s
#    Fix: increase pool size OR reduce query time OR add read replica
# 2. Connection leak: code takes connection, throws exception before returning it
#    Fix: leak-detection-threshold catches these; use @Transactional or try-with-resources
# 3. Too many connections: total app connections > DB max_connections
#    Fix: use a connection proxy like PgBouncer/ProxySQL in front of DB
