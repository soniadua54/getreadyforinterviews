Key terms
The problem before CI/CD
Before automated pipelines, teams would manually build, test, and deploy. A release might take days: developer finishes code → manually runs tests → creates a ZIP → emails it to ops → ops manually copies to server → cross fingers. This led to "integration hell" — multiple developers merging weeks of work simultaneously, each breaking each other's code.

Continuous Integration (CI)

Automatically build and test every code commit. Detects broken code within minutes, not days. Goal: merge small changes often, never let main branch break.

Continuous Delivery (CD)

Automatically deploy every successful build to staging. Production deploy is still a manual button push. Goal: always have a deployable artifact ready.

Continuous Deployment

Like CD but production deploy is also automatic — no manual gate. Used by Netflix, Amazon (thousands of deploys per day). Requires extremely mature testing and monitoring.

GitOps

Git repo is the single source of truth for infrastructure AND app state. What's in Git = what's in production. ArgoCD watches Git, auto-syncs to K8s. You never run kubectl manually in production.

Full CI/CD pipeline — from git push to production
Developer pushes code to feature branch ↓ [GitHub / GitLab / Bitbucket] ↓ pull request opened [CI Pipeline triggered] Stage 1: Build ├── Compile (mvn package / npm ci) ├── Static analysis (SonarQube, Checkstyle) └── SAST security scan (Snyk, OWASP) Stage 2: Test ├── Unit tests (JUnit, Jest) ├── Integration tests (TestContainers) └── Code coverage gate (must be > 80%) Stage 3: Package ├── Docker image build ├── Image vulnerability scan (Trivy) └── Push to registry (ECR / GCR / Docker Hub) ↓ merge to main [CD Pipeline triggered] Stage 4: Deploy to Staging ├── Helm upgrade / kubectl apply ├── Smoke tests └── Automated E2E tests (Cypress / Playwright) ↓ manual approval gate Stage 5: Deploy to Production ├── Blue/green or canary deploy ├── Health check verification ├── Rollback on failure (automatic) └── Notify team (Slack, PagerDuty)
TermWhat it means
PipelineSequence of automated stages: build → test → deploy
ArtifactOutput of a build: JAR file, Docker image, npm package
Stage / JobOne step in the pipeline: a script that runs and succeeds or fails
Runner / AgentMachine that executes pipeline jobs (GitHub-hosted or self-hosted)
TriggerEvent that starts the pipeline: push, PR, cron, manual
EnvironmentDev → Staging → Production — each with own config and deploy
Blue/Green deployTwo identical environments. Switch traffic from blue to green. Instant rollback = switch back.
Canary deploySend 5% of traffic to new version. If metrics OK, gradually increase to 100%.
Feature flagsDeploy code to production but hide behind a flag. Enable for 1% of users first.
RollbackRedeploy previous version automatically when health checks fail.
GitOpsInfrastructure state is stored in Git. Tools like ArgoCD sync K8s to match Git state.
Helm chartPackage manager for Kubernetes. Template-based K8s manifests with values override per environment.

GitHub Actions structure
Workflows live in .github/workflows/*.yml. Each workflow has triggers, jobs, and steps. Jobs run in parallel by default; use needs to sequence them.
# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [ main, develop ]      # trigger on push to these branches
  pull_request:
    branches: [ main ]               # trigger on PR targeting main
  workflow_dispatch:                 # allow manual trigger

jobs:
  build:
    runs-on: ubuntu-latest           # GitHub-hosted runner
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4    # clone the repo
      
      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
      
      - name: Run tests
        run: ./mvnw test
      
      - name: Build JAR
        run: ./mvnw package -DskipTests

  deploy-staging:
    needs: build                     # only runs if build succeeds
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'  # only on main branch
    
    steps:
      - name: Deploy to staging
        run: echo "deploy step here"
Key concepts: uses: runs a pre-built Action from Marketplace. run: runs a shell command directly. needs: creates dependency between jobs. if: conditionally runs a job.
# Complete Java/Spring Boot CI/CD pipeline
name: Java CI/CD

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

env:
  JAVA_VERSION: '17'
  IMAGE_NAME: hotel-service
  ECR_REGISTRY: ${{ secrets.AWS_ACCOUNT_ID }}.dkr.ecr.us-east-1.amazonaws.com

jobs:
  test:
    name: Test
    runs-on: ubuntu-latest
    
    services:
      postgres:                        # spin up DB for integration tests
        image: postgres:15-alpine
        env:
          POSTGRES_DB: testdb
          POSTGRES_USER: test
          POSTGRES_PASSWORD: test
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-retries 5
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Set up JDK
        uses: actions/setup-java@v4
        with:
          java-version: ${{ env.JAVA_VERSION }}
          distribution: temurin
          cache: maven                 # cache ~/.m2 between runs
      
      - name: Run tests
        run: ./mvnw verify            # compile + test + integration tests
        env:
          SPRING_DATASOURCE_URL: jdbc:postgresql://localhost:5432/testdb
          SPRING_DATASOURCE_USERNAME: test
          SPRING_DATASOURCE_PASSWORD: test
      
      - name: Upload test report
        uses: actions/upload-artifact@v4
        if: always()                  # upload even on failure
        with:
          name: test-results
          path: target/surefire-reports/

  build-and-push:
    name: Build Docker Image
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    outputs:
      image-tag: ${{ steps.meta.outputs.version }}  # pass tag to next job
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region: us-east-1
      
      - name: Login to ECR
        uses: aws-actions/amazon-ecr-login@v2
      
      - name: Docker metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.ECR_REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha,prefix=,format=short     # git short SHA as tag
            type=raw,value=latest,enable=true
      
      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy-staging:
    name: Deploy to Staging
    needs: build-and-push
    runs-on: ubuntu-latest
    environment: staging             # requires environment approval if configured
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Deploy via Helm
        run: |
          aws eks update-kubeconfig --region us-east-1 --name staging-cluster
          helm upgrade --install hotel-service ./helm/hotel-service \
            --namespace hotel \
            --set image.tag=${{ needs.build-and-push.outputs.image-tag }} \
            --set environment=staging \
            --wait --timeout 5m
      
      - name: Smoke test
        run: |
          sleep 10
          curl -f https://staging.hotel-service.com/actuator/health || exit 1

  deploy-production:
    name: Deploy to Production
    needs: deploy-staging
    runs-on: ubuntu-latest
    environment: production           # REQUIRES manual approval in GitHub UI
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Deploy to production
        run: |
          aws eks update-kubeconfig --region us-east-1 --name prod-cluster
          helm upgrade hotel-service ./helm/hotel-service \
            --namespace hotel \
            --set image.tag=${{ needs.build-and-push.outputs.image-tag }} \
            --set environment=production \
            --wait --timeout 10m
      
      - name: Notify Slack on success
        if: success()
        uses: slackapi/slack-github-action@v1
        with:
          payload: '{"text":"✅ hotel-service deployed to production"}'
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
      
      - name: Rollback on failure
        if: failure()
        run: helm rollback hotel-service --namespace hotel

# Reusable Docker build workflow
name: Docker Build

on:
  workflow_call:                       # reusable workflow — called from other workflows
    inputs:
      image-name:
        required: true
        type: string
    outputs:
      image-tag:
        value: ${{ jobs.build.outputs.tag }}

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      tag: ${{ steps.meta.outputs.version }}
    
    steps:
      - uses: actions/checkout@v4
      
      - name: Set up QEMU             # for multi-platform builds (arm64 + amd64)
        uses: docker/setup-qemu-action@v3
      
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3
      
      - name: Docker metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ inputs.image-name }}
          tags: |
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha
      
      - name: Build and push multi-platform
        uses: docker/build-push-action@v5
        with:
          platforms: linux/amd64,linux/arm64    # Apple Silicon + x86
          push: ${{ github.event_name != 'pull_request' }}
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
      
      - name: Scan image for vulnerabilities
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: ${{ inputs.image-name }}:${{ steps.meta.outputs.version }}
          severity: CRITICAL,HIGH
          exit-code: 1                 # fail if critical vulns found

GitOps flow with ArgoCD
Developer pushes code → GitHub Actions builds image → pushes to ECR ↓ GitHub Actions updates image tag in git repo (gitops-repo/apps/hotel-service/values.yaml) ↓ ArgoCD continuously watches gitops-repo ↓ ArgoCD detects drift (current K8s ≠ Git state) ↓ ArgoCD syncs: applies new deployment to K8s ↓ K8s rolls out new pods with new image tag Key principle: You NEVER run kubectl apply manually in production. Git is the only way to change what's in production. Rollback = revert the git commit, ArgoCD auto-syncs.
Why GitOps beats imperative deploy: Every change is a git commit — full audit trail, easy rollback, peer-reviewed changes, no snowflake servers. At UKG, TRex manages tenant config in Git for the same reason — the config file IS the state.

      valueFiles:
        - values-production.yaml

Rolling update — K8s default
Replace pods gradually. New pods are created before old ones are terminated. Zero downtime. If new pods fail health checks, K8s pauses the rollout and leaves old pods running.
# Deployment with rolling update strategy
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hotel-service
spec:
  replicas: 6
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 2          # max extra pods during update (6 + 2 = 8 max)
      maxUnavailable: 0    # never remove old pod before new one is ready
  template:
    spec:
      containers:
      - name: hotel-service
        image: hotel-service:v2
        readinessProbe:    # K8s waits for this before routing traffic
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
          failureThreshold: 3

# Monitor rollout
kubectl rollout status deployment/hotel-service -n hotel
kubectl rollout history deployment/hotel-service -n hotel

# Rollback instantly
kubectl rollout undo deployment/hotel-service -n hotel
kubectl rollout undo deployment/hotel-service -n hotel --to-revision=3
Blue/Green deployment
State: [Blue: v1 - LIVE] [Green: v2 - STAGED] ↑ receives traffic ↑ running but no traffic Deploy: Deploy v2 to Green environment Run smoke tests against Green Switch: Update Service selector: selector: version: green ← instant traffic switch Result: [Blue: v1 - STANDBY] [Green: v2 - LIVE] ↑ receives all traffic Rollback: selector: version: blue ← instant (Blue still running)
# Blue deployment (currently live)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hotel-service-blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: hotel-service
      version: blue
  template:
    metadata:
      labels:
        app: hotel-service
        version: blue     # ← differentiator
    spec:
      containers:
      - name: hotel-service
        image: hotel-service:v1

# Service points to blue (current live)
apiVersion: v1
kind: Service
metadata:
  name: hotel-service
spec:
  selector:
    app: hotel-service
    version: blue         # ← switch this to 'green' to flip traffic

# After deploying green and testing, switch traffic:
kubectl patch service hotel-service -n hotel \
  -p '{"spec":{"selector":{"version":"green"}}}'

# Rollback is instant — just switch back:
kubectl patch service hotel-service -n hotel \
  -p '{"spec":{"selector":{"version":"blue"}}}'
# Canary: send 10% of traffic to new version
# Technique: run more stable replicas than canary replicas
# 9 stable + 1 canary = 10% of traffic to canary

# Stable deployment (9 replicas = 90% traffic)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hotel-service-stable
spec:
  replicas: 9
  selector:
    matchLabels:
      app: hotel-service
      track: stable
  template:
    metadata:
      labels:
        app: hotel-service
        track: stable
    spec:
      containers:
      - image: hotel-service:v1

# Canary deployment (1 replica = ~10% traffic)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hotel-service-canary
spec:
  replicas: 1
  selector:
    matchLabels:
      app: hotel-service
      track: canary
  template:
    metadata:
      labels:
        app: hotel-service
        track: canary
    spec:
      containers:
      - image: hotel-service:v2       # new version

# Service selects BOTH (by shared label)
apiVersion: v1
kind: Service
spec:
  selector:
    app: hotel-service                # matches BOTH stable and canary
                                      # K8s load balances 9:1 ratio

# If canary metrics look good: scale up canary, scale down stable
kubectl scale deployment hotel-service-canary --replicas=9 -n hotel
kubectl scale deployment hotel-service-stable --replicas=1 -n hotel
# Eventually remove stable:
kubectl delete deployment hotel-service-stable -n hotel

# Sealed Secrets — encrypt K8s secrets so they're safe to commit to git
# Problem: K8s Secret is just base64, NOT encrypted. Don't commit to git.
# Solution: SealedSecret is encrypted with cluster key — safe to commit.

# Install Sealed Secrets controller
helm install sealed-secrets \
  sealed-secrets/sealed-secrets \
  --namespace kube-system

# Create a sealed secret from a plain secret
kubectl create secret generic hotel-db-secret \
  --from-literal=password=mysecret \
  --dry-run=client -o yaml | \
  kubeseal --format yaml > hotel-db-sealed.yaml

# hotel-db-sealed.yaml — safe to commit to git!
apiVersion: bitnami.com/v1alpha1
kind: SealedSecret
metadata:
  name: hotel-db-secret
spec:
  encryptedData:
    password: AgA9Xk2...long_encrypted_string...==

# Apply to cluster (controller decrypts automatically)
kubectl apply -f hotel-db-sealed.yaml

# Controller creates a regular K8s Secret from the SealedSecret
kubectl get secret hotel-db-secret -o jsonpath='{.data.password}' | base64 -d

Test pyramid — what runs where in pipeline
[E2E Tests] ← runs post-deploy on staging / few \ Playwright/Cypress / Slow, fragile \ Tests full user journey ──────────────────── [Integration Tests] ← runs in CI (TestContainers) / Database, API calls \ Tests your code + real DB / Medium speed \ ────────────────────────────── [Unit Tests] ← runs in CI, fast / Mock everything external \ JUnit + Mockito / Fast, isolated, many \ Tests one class/method ────────────────────────────────────── [Contract Tests] ← consumer-driven Consumer defines expected API shape Pact framework Provider verifies it satisfies contract [Load Tests] ← runs in staging k6 / Gatling / JMeter Before production deploy
// Integration tests with TestContainers — real DB in CI
@SpringBootTest
@Testcontainers
class BookingServiceIntegrationTest {

    @Container
    static PostgreSQLContainer<?> postgres =
        new PostgreSQLContainer<>("postgres:15-alpine")
            .withDatabaseName("testdb")
            .withUsername("test")
            .withPassword("test");

    @Container
    static GenericContainer<?> redis =
        new GenericContainer<>("redis:7-alpine").withExposedPorts(6379);

    @DynamicPropertySource
    static void configure(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", postgres::getJdbcUrl);
        registry.add("spring.datasource.username", postgres::getUsername);
        registry.add("spring.redis.host", redis::getHost);
        registry.add("spring.redis.port", () -> redis.getMappedPort(6379).toString());
    }

    @Autowired BookingService bookingService;

    @Test
    void createBooking_persistsToDatabase() {
        CreateBookingRequest req = CreateBookingRequest.builder()
            .userId("user-1").roomId("room-101").checkIn(LocalDate.now())
            .checkOut(LocalDate.now().plusDays(3)).build();

        Booking created = bookingService.create(req);

        assertThat(created.getId()).isNotNull();
        assertThat(created.getStatus()).isEqualTo(BookingStatus.CONFIRMED);
    }
}
// Consumer-driven contract testing with Pact
// Consumer: notification-service defines what it expects from booking-service API
// Provider: booking-service verifies it actually delivers that contract

// 1. Consumer side — defines the contract
@ExtendWith(PactConsumerTestExt.class)
@PactTestFor(providerName = "booking-service")
class NotificationServiceConsumerTest {

    @Pact(consumer = "notification-service")
    public RequestResponsePact getBookingPact(PactDslWithProvider builder) {
        return builder
            .given("booking with id 123 exists")
            .uponReceiving("a request for booking 123")
            .path("/api/v1/bookings/123")
            .method("GET")
            .willRespondWith()
            .status(200)
            .body(new PactDslJsonBody()
                .stringType("id", "123")
                .stringType("userId", "user-1")
                .stringType("status", "CONFIRMED"))
            .toPact();
    }
}

        context.verifyInteraction();

# DORA metrics — how to measure pipeline health
# (The 4 key DevOps metrics every staff engineer should know)

# 1. Deployment Frequency — how often you deploy to production
#    Elite: multiple times per day | Good: weekly | Poor: monthly

# 2. Lead Time for Changes — commit to production
#    Elite: < 1 hour | Good: < 1 day | Poor: months

# 3. Change Failure Rate — % of deploys causing incidents
#    Elite: 0-15% | Poor: > 45%

# 4. Mean Time to Restore (MTTR) — incident to resolution
#    Elite: < 1 hour | Poor: > 1 week

# Track in pipeline:
- name: Record deploy time
  run: |
    COMMIT_TIME=$(git log -1 --format=%ct HEAD)
    DEPLOY_TIME=$(date +%s)
    LEAD_TIME=$((DEPLOY_TIME - COMMIT_TIME))
    echo "Lead time: ${LEAD_TIME}s" >> metrics.log
