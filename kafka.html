Core Concepts — What Kafka Is and Why
Topics, partitions, offsets, brokers — the mental model
Beginner▼
Why Kafka
Architecture
Kafka vs alternatives
The problem Kafka solves
When a booking is confirmed, five things need to happen: send email, send SMS, update loyalty points, sync analytics, trigger voucher generation. If you do all five synchronously in the booking API — the response takes 2 seconds and any one failure fails the whole booking. If you call each service directly (HTTP) — you have tight coupling. If the email service is down, your booking service breaks.
Kafka decouples producers from consumers. Booking service publishes one event to Kafka. Email service, SMS service, analytics service — all subscribe independently. Booking service finishes in milliseconds. If email service is down, the message waits in Kafka until it recovers. No coupling, no data loss.

Without Kafka

Booking → calls Email, SMS, Analytics directly. One failure = cascading failures. Tight coupling. Adding a new consumer requires changing booking code.

With Kafka

Booking → publishes to Kafka. Consumers subscribe independently. New consumer = zero changes to booking service. Each service scales independently.

// Producer — send events from Booking Service
@Service
public class BookingEventProducer {
    private final KafkaTemplate<String, BookingEvent> kafkaTemplate;

    public void publishBookingCreated(Booking booking) {
        BookingEvent event = BookingEvent.builder()
            .bookingId(booking.getId())
            .userId(booking.getUserId())
            .roomId(booking.getRoomId())
            .type("BOOKING_CREATED")
            .occurredAt(Instant.now())
            .build();

        // Key = userId → all events for same user go to same partition (ordered)
        var record = new ProducerRecord<>("booking.events", booking.getUserId(), event);
        record.headers().add("eventType", "BOOKING_CREATED".getBytes());

        kafkaTemplate.send(record)
            .whenComplete((result, ex) -> {
                if (ex != null) {
                    log.error("Failed to publish bookingId={}", booking.getId(), ex);
                    // could also push to outbox table for retry
                } else {
                    log.info("Published bookingId={} partition={} offset={}",
                        booking.getId(),
                        result.getRecordMetadata().partition(),
                        result.getRecordMetadata().offset());
                }
            });
    }
}
// Consumer — Notification Service reads booking events
@Component
public class BookingEventConsumer {

    @KafkaListener(
        topics = "booking.events",
        groupId = "notification-service",
        concurrency = "3"  // 3 threads = can consume 3 partitions in parallel
    )
    public void handle(
            BookingEvent event,
            @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
            @Header(KafkaHeaders.OFFSET) long offset,
            Acknowledgment ack) {  // manual ack

        log.info("Processing event={} partition={} offset={}",
            event.getBookingId(), partition, offset);

        try {
            notificationService.sendConfirmation(event);
            ack.acknowledge();  // commit offset ONLY after successful processing
        } catch (RetryableException e) {
            // don't ack — Kafka will redeliver
            log.warn("Retryable error, will retry: {}", e.getMessage());
        } catch (NonRetryableException e) {
            // move to DLT (dead letter topic) instead of infinite retry
            log.error("Non-retryable error, sending to DLT", e);
            ack.acknowledge();  // ack so we don't redeliver
            dltProducer.send(event, e);
        }
    }
}

// Error handling with retry + DLT
@Bean
public DefaultErrorHandler errorHandler(KafkaTemplate<?,?> template) {
    // Send to booking.events.DLT after 3 retries
    DeadLetterPublishingRecoverer recoverer =
        new DeadLetterPublishingRecoverer(template,
            (r, e) -> new TopicPartition(r.topic() + ".DLT", r.partition()));

    // Retry with exponential backoff: 1s → 2s → 4s → DLT
    ExponentialBackOff backoff = new ExponentialBackOff(1000L, 2.0);
    backoff.setMaxAttempts(3);

    DefaultErrorHandler handler = new DefaultErrorHandler(recoverer, backoff);
    // Don't retry these — they'll never succeed
    handler.addNotRetryableExceptions(
        JsonParseException.class,
        IllegalArgumentException.class
    );
    return handler;
}
# application.yml — Kafka config
spring:
  kafka:
    bootstrap-servers: kafka-1:9092,kafka-2:9092,kafka-3:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
      acks: all          # wait for ALL replicas to acknowledge (strongest)
      retries: 3
      properties:
        enable.idempotence: true    # exactly-once from producer side
        max.in.flight.requests.per.connection: 1
    consumer:
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
      group-id: notification-service
      auto-offset-reset: earliest   # start from beginning if no committed offset
      enable-auto-commit: false     # NEVER use auto-commit — manual ack only
      properties:
        spring.json.trusted.packages: "com.yourcompany.events"
    listener:
      ack-mode: MANUAL_IMMEDIATE    # manual acknowledgment
      concurrency: 3                # 3 threads per listener container

# Producer acks explained:
# acks=0  → fire and forget (fastest, data loss possible)
# acks=1  → leader acknowledges (default, leader failure = data loss)
# acks=all → all in-sync replicas ack (slowest, no data loss)

Partition Key Strategy — The Most Important Design Decision
Wrong partition key = no ordering, hot partitions, broken consumers
Intermediate▼
Why partition key matters
Messages with the same key always go to the same partition. Kafka guarantees ordering within a partition, not across partitions. So your key must be the entity whose events must be ordered.
Choosing the right partition key
Wrong key: no key (round-robin) Event 1 (BOOKING_CREATED) → Partition 0 Event 2 (BOOKING_CANCELLED) → Partition 1 Consumer processes Partition 1 first → CANCELLED before CREATED → broken! Wrong key: bookingId BOOKING_CREATED and PAYMENT_SUCCESS for booking-123 → different partitions If you need them ordered → wrong choice Right key for booking events: userId All events for user-456 → same partition → always in order Consumers see: BOOKING_CREATED → PAYMENT_SUCCESS → BOOKING_CONFIRMED ✓ Right key for hotel inventory: hotelId All inventory changes for hotel-789 → same partition → no race conditions Hot partition problem: If key distribution is uneven (one userId has millions of bookings) → one partition gets all the traffic → that consumer falls behind Fix: append random suffix for high-volume keys: "userId-123-0", "userId-123-1"
// Partition key design in code
// Booking events: partition by userId (order per user matters)
kafkaTemplate.send("booking.events", booking.getUserId(), event);

// Hotel inventory: partition by hotelId (order per hotel matters)
kafkaTemplate.send("hotel.inventory", update.getHotelId(), update);

// Notification fanout: no ordering needed → null key (round-robin)
kafkaTemplate.send("notification.send", null, notification);

// Hot partition fix: spread high-volume keys
int NUM_VIRTUAL_PARTITIONS = 10;
String key = userId + "-" + (Math.abs(eventId.hashCode()) % NUM_VIRTUAL_PARTITIONS);
kafkaTemplate.send("booking.events", key, event);
// NOTE: this breaks strict per-user ordering, acceptable if just need distribution

Consumer Groups — Parallel Processing & Scaling
How Kafka distributes work across multiple consumers
Intermediate▼
Consumer group mechanics
Topic "booking.events" — 4 partitions Scenario 1: 1 consumer in group consumer-1 → reads ALL 4 partitions (sequential) Scenario 2: 4 consumers in group (optimal) consumer-1 → Partition 0 consumer-2 → Partition 1 consumer-3 → Partition 2 consumer-4 → Partition 3 Scenario 3: 6 consumers, 4 partitions (wasteful) consumer-1 → Partition 0 consumer-2 → Partition 1 consumer-3 → Partition 2 consumer-4 → Partition 3 consumer-5 → IDLE (no partition to read) consumer-6 → IDLE Rule: consumers > partitions = idle consumers, no benefit Scenario 4: 2 separate consumer groups Group "notification-service": consumer-1 reads all 4 partitions Group "analytics-service": consumer-1 reads all 4 partitions Both get ALL messages independently (different offsets per group) Rebalancing: when consumer joins/leaves, partitions reassigned During rebalance: consumers PAUSE (stop-the-world) Minimize rebalances: use static group membership, tune session.timeout.ms
# Key consumer config decisions
// session.timeout.ms (default 45s)
// How long before broker considers consumer dead
// If processing takes longer than this → rebalance triggered
// For slow processors: increase to 300s

// max.poll.interval.ms (default 5m)
// Max time between poll() calls
// If your business logic takes > 5min → increase this

// max.poll.records (default 500)
// Messages per poll() call
// Reduce if processing is slow (process fewer at a time)

// Static group membership — reduces rebalances on restarts
// group.instance.id = "consumer-pod-1" (unique per pod)
// Kafka waits session.timeout before reassigning partition
// Great for rolling deployments — partitions stay with same pods

# In Spring Boot:
spring:
  kafka:
    consumer:
      properties:
        max.poll.interval.ms: 300000    # 5 min max processing time
        max.poll.records: 100           # smaller batches
        session.timeout.ms: 60000       # 60s before considered dead
        group.instance.id: ${POD_NAME} # static membership (K8s env var)

Delivery Guarantees — At-least-once vs Exactly-once
The Staff-level question: how do you prevent duplicate processing?
Advanced▼
Three delivery semantics
At-most-once (fire and forget) Producer sends, doesn't wait for ack acks=0 Result: fast, but messages CAN be lost Use: metrics, logs where some loss is OK At-least-once (default for most systems) Producer retries until broker acks Consumer processes, then commits offset Result: NO data loss, but possible DUPLICATE processing If consumer crashes after processing but before committing offset → message redelivered → processed again Use: most business events (with idempotent consumers) Exactly-once (EOS — Exactly Once Semantics) Kafka 0.11+ with transactions Producer: enable.idempotence=true + transactional.id Consumer: read_committed isolation level Result: no loss, no duplicates Cost: ~10-20% throughput reduction Use: financial transactions, inventory updates
// Idempotent consumer — the practical "exactly-once" for most cases
// At-least-once delivery + idempotent processing = effectively exactly-once

@KafkaListener(topics = "payment.events", groupId = "payment-processor")
public void handlePayment(PaymentEvent event, Acknowledgment ack) {
    // Check if already processed (idempotency key)
    if (processedEventRepo.existsById(event.getEventId())) {
        log.info("Duplicate event {}, skipping", event.getEventId());
        ack.acknowledge();
        return; // already done — safe to skip
    }

    // Process + mark as processed atomically
    @Transactional
    walletService.credit(event.getUserId(), event.getAmount());
    processedEventRepo.save(new ProcessedEvent(event.getEventId()));

    ack.acknowledge();
}

// Kafka Transactions — true exactly-once (read-process-write atomic)
// Producer config:
// transactional.id = "payment-service-0"  (unique per producer instance)
// enable.idempotence = true

@Bean
public KafkaTransactionManager kafkaTransactionManager(...) {
    return new KafkaTransactionManager(producerFactory);
}

@Transactional("kafkaTransactionManager")
public void processAndPublish(PaymentEvent in) {
    // Read from Kafka + write to Kafka — atomic transaction
    // Either both happen or neither
    walletService.credit(in.getUserId(), in.getAmount());
    kafkaTemplate.send("wallet.credited", in.getUserId(), new WalletCreditedEvent(in));
}

Schema Registry & Avro
Evolving message schemas without breaking consumers
Advanced▼
JSON schemas can change without warning — a producer adds a field, consumers break. Schema Registry enforces contract between producers and consumers. Avro schemas are stored centrally; producers/consumers auto-validate against them. Schema evolution rules (backward/forward compatibility) prevent breaking changes.
# Schema Registry — Confluent (or AWS Glue for AWS)
# Avro schema for BookingEvent
{
  "type": "record",
  "name": "BookingEvent",
  "namespace": "com.yourcompany.events",
  "fields": [
    {"name": "bookingId", "type": "string"},
    {"name": "userId", "type": "string"},
    {"name": "roomId", "type": "string"},
    {"name": "checkIn", "type": "string"},
    {"name": "totalPrice", "type": "double"},
    {"name": "currency", "type": {"type": "enum", "name": "Currency", "symbols": ["USD","EUR","GBP"]}},
    {"name": "couponCode", "type": ["null", "string"], "default": null}  // optional field
  ]
}

# Schema evolution rules:
# BACKWARD compatible: consumers with new schema can read old messages
#   → can ADD optional fields (with default)
#   → can REMOVE fields consumers don't use
# FORWARD compatible: consumers with old schema can read new messages
# FULL compatible: both backward + forward
#
# NEVER do (breaks compatibility):
#   → rename a field
#   → change a field type
#   → remove a required field

# application.yml with Schema Registry
spring:
  kafka:
    producer:
      value-serializer: io.confluent.kafka.serializers.KafkaAvroSerializer
    consumer:
      value-deserializer: io.confluent.kafka.serializers.KafkaAvroDeserializer
    properties:
      schema.registry.url: http://schema-registry:8081
      specific.avro.reader: true   # deserialize to generated Java class
