K8s Architecture — What Each Component Does
Control plane, worker nodes, etcd, scheduler, kubelet
Beginner▼
Architecture
Core objects
Kubernetes cluster architecture
CONTROL PLANE (master node) ├── kube-apiserver ← all kubectl commands go here. REST API. ├── etcd ← distributed key-value store. ALL cluster state lives here. ├── kube-scheduler ← decides WHICH node to run a new pod on │ factors: CPU/memory available, node affinity, taints ├── kube-controller-manager ← runs control loops: │ ├── ReplicaSet controller (ensures desired # pods running) │ ├── Deployment controller (rolling updates) │ └── Endpoint controller (updates Service endpoints) └── cloud-controller-manager ← talks to cloud provider (AWS/GCP/Azure) WORKER NODES (where your code runs) ├── kubelet ← agent on each node. Runs pods. Reports to control plane. ├── kube-proxy ← network proxy. Maintains iptables rules for Services. └── Container runtime ← runs containers (containerd, CRI-O) └── Pods ← your application containers run here kubectl → API Server → etcd (stored) → Scheduler (placed) → kubelet (started)
ObjectWhat it isAnalogy
PodSmallest deployable unit. 1+ containers sharing network + storage. Ephemeral — gets a new IP on restart.A running process (or a few tightly coupled processes)
ReplicaSetEnsures N copies of a pod are always running. Replaces crashed pods automatically.Process manager (like PM2) but cluster-wide
DeploymentManages ReplicaSets. Handles rolling updates, rollbacks. What you deploy in practice.Version-aware process manager
ServiceStable IP + DNS name for a set of pods. Load balances across them. Pods come and go, Service stays.Load balancer
IngressHTTP/HTTPS routing. Maps domains/paths to Services. Requires Ingress Controller (nginx, Traefik).Reverse proxy / API gateway
ConfigMapNon-sensitive config data. Mounted as env vars or files into pods.application.yml injected at runtime
SecretSensitive data (base64 encoded, not encrypted by default). Mounted same as ConfigMap.Encrypted environment variables
PersistentVolumeStorage that outlives a pod. Backed by EBS, NFS, GCS etc.External hard drive
NamespaceVirtual cluster. Logical isolation of resources. hotel-service, monitoring, kube-system.AWS account / GCP project
HPAHorizontal Pod Autoscaler. Scales replica count based on CPU/memory/custom metrics.Auto-scaling group

# deployment.yaml — production-grade hotel service
apiVersion: apps/v1
kind: Deployment
metadata:
  name: hotel-service
  namespace: hotel
  labels:
    app: hotel-service
    version: v1.2.3
spec:
  replicas: 3
  selector:
    matchLabels:
      app: hotel-service
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # one extra pod during update
      maxUnavailable: 0    # never kill old pod before new is ready
  template:
    metadata:
      labels:
        app: hotel-service
        version: v1.2.3
    spec:
      containers:
      - name: hotel-service
        image: 123456789.dkr.ecr.us-east-1.amazonaws.com/hotel-service:v1.2.3
        ports:
        - containerPort: 8080
        
        env:
        - name: SPRING_PROFILES_ACTIVE
          value: "production"
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: hotel-db-secret  # from Secret object
              key: password
        - name: APP_CONFIG
          valueFrom:
            configMapKeyRef:
              name: hotel-config     # from ConfigMap
              key: max-connections
        
        resources:
          requests:
            cpu: 250m               # 0.25 CPU core
            memory: 512Mi
          limits:
            cpu: 1000m              # max 1 CPU core
            memory: 1Gi             # OOM killed if exceeded
        
        # Probes — K8s uses these to manage pod lifecycle
        startupProbe:               # wait for app to start (Spring Boot takes 30-60s)
          httpGet:
            path: /actuator/health
            port: 8080
          failureThreshold: 30      # try 30 times
          periodSeconds: 10         # every 10s = max 300s startup
        
        livenessProbe:              # restart pod if fails
          httpGet:
            path: /actuator/health/liveness
            port: 8080
          initialDelaySeconds: 0    # startupProbe handles initial delay
          periodSeconds: 30
          failureThreshold: 3
        
        readinessProbe:             # remove from Service if fails (don't send traffic)
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          periodSeconds: 10
          failureThreshold: 3
      
      # Graceful shutdown
      terminationGracePeriodSeconds: 60
      
      # Spread pods across nodes (avoid all on same node)
      topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: kubernetes.io/hostname
        whenUnsatisfiable: DoNotSchedule
        labelSelector:
          matchLabels:
            app: hotel-service
# service.yaml — stable IP + DNS for pods
apiVersion: v1
kind: Service
metadata:
  name: hotel-service
  namespace: hotel
spec:
  selector:
    app: hotel-service          # routes to pods with this label
  ports:
  - port: 80                    # Service port
    targetPort: 8080            # Container port
    protocol: TCP
  type: ClusterIP               # internal only (ClusterIP, NodePort, LoadBalancer)
# DNS: hotel-service.hotel.svc.cluster.local resolves to Service IP

---
# ingress.yaml — HTTP routing from outside cluster
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: hotel-service-ingress
  namespace: hotel
  annotations:
    nginx.ingress.kubernetes.io/rate-limit: "100"
    nginx.ingress.kubernetes.io/rate-limit-window: "1m"
    cert-manager.io/cluster-issuer: "letsencrypt-prod"   # auto TLS
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - api.hotelservice.com
    secretName: hotel-tls-cert
  rules:
  - host: api.hotelservice.com
    http:
      paths:
      - path: /api/v1/hotels
        pathType: Prefix
        backend:
          service:
            name: hotel-service
            port:
              number: 80
      - path: /api/v1/bookings
        pathType: Prefix
        backend:
          service:
            name: booking-service
            port:
              number: 80

Autoscaling — HPA & VPA
Scale pods based on CPU, memory, and custom metrics
Intermediate▼ 
# HPA — Horizontal Pod Autoscaler
# Scales replica count up/down based on metrics
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: hotel-service-hpa
  namespace: hotel
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: hotel-service
  minReplicas: 3
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70     # scale up if avg CPU > 70%
  - type: Resource
    resource:
      name: memory
      target:
        type: AverageValue
        averageValue: 700Mi        # scale up if avg memory > 700Mi
  # Custom metric — scale based on requests per second
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
      target:
        type: AverageValue
        averageValue: "100"        # scale up if > 100 req/sec per pod
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60   # don't scale up too fast
    scaleDown:
      stabilizationWindowSeconds: 300  # wait 5 min before scaling down

# Check HPA status
kubectl get hpa hotel-service-hpa -n hotel
kubectl describe hpa hotel-service-hpa -n hotel

# Typical output:
# NAME                   REFERENCE                TARGETS         MINPODS   MAXPODS   REPLICAS
# hotel-service-hpa      Deployment/hotel-service 65%/70%         3         20        5

Debug toolkit
CrashLoopBackOff — diagnosis flow
pod status = CrashLoopBackOff Step 1: Get pod name kubectl get pods -n hotel Step 2: Check events (most useful first look) kubectl describe pod hotel-service-7d4b5f9c8-xk2jn -n hotel Look for: Events section at the bottom Common causes: OOMKilled, ImagePullBackOff, probe failure Step 3: Check container logs kubectl logs hotel-service-7d4b5f9c8-xk2jn -n hotel kubectl logs hotel-service-7d4b5f9c8-xk2jn -n hotel --previous ← crashed container logs Step 4: Check if it's memory kubectl describe pod ... | grep -A5 "Last State" "OOMKilled" → increase memory limit Step 5: Check if startup probe failing Spring Boot takes 60s to start. If startupProbe fails in 30s → crash. Fix: increase failureThreshold * periodSeconds Step 6: Shell into pod (if it's staying up briefly) kubectl exec -it hotel-service-xxx -n hotel -- /bin/sh
# Pod stuck in Pending — can't be scheduled
kubectl describe pod hotel-service-xxx -n hotel

# Common reasons:
# 1. Insufficient resources
# Events: "0/3 nodes are available: 3 Insufficient cpu"
# Fix: reduce resource requests OR scale up node pool
kubectl top nodes                  # check node capacity

# 2. PersistentVolumeClaim not bound
# Events: "pod has unbound immediate PersistentVolumeClaims"
kubectl get pvc -n hotel           # check PVC status
kubectl describe pvc my-pvc -n hotel

# 3. Node affinity / taint mismatch
# Events: "0/3 nodes are available: 3 node(s) had untolerated taint"
kubectl describe node node-1 | grep Taints
# Add toleration to pod spec or remove taint from node

# 4. Image pull failure
# Events: "Failed to pull image: ... access denied"
kubectl get secret ecr-credentials -n hotel
# Fix: create imagePullSecret with ECR credentials
kubectl create secret docker-registry ecr-credentials \
  --docker-server=123456789.dkr.ecr.us-east-1.amazonaws.com \
  --docker-username=AWS \
  --docker-password=$(aws ecr get-login-password) \
  -n hotel
# ── ESSENTIAL DEBUG COMMANDS ──

# Cluster overview
kubectl get all -n hotel           # everything in namespace
kubectl get events -n hotel --sort-by='.lastTimestamp' | tail -20

kubectl logs <pod-name> -n hotel -f         # stream logs

# ServiceMonitor — tells Prometheus to scrape your service
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: hotel-service
  namespace: hotel
spec:
  selector:
    matchLabels:
      app: hotel-service
  endpoints:
  - port: http
    path: /actuator/prometheus
    interval: 30s

# Key metrics to monitor for a booking service:
# HTTP layer
http_server_requests_seconds_count{uri="/api/v1/bookings"}  # request rate
http_server_requests_seconds_bucket{le="0.2"}               # p80 latency < 200ms

# JVM
jvm_memory_used_bytes{area="heap"}        # heap usage
jvm_gc_pause_seconds_sum                   # GC time
jvm_threads_live_threads                   # thread count

# Business metrics (custom)
bookings_created_total                     # bookings per minute
payment_processing_duration_seconds        # p99 payment latency

# PrometheusRule — define alerts
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: hotel-service-alerts
spec:
  groups:
  - name: hotel-service
    rules:
    - alert: HighErrorRate
      expr: |
        rate(http_server_requests_seconds_count{status=~"5.."}[5m]) /
        rate(http_server_requests_seconds_count[5m]) > 0.05
      for: 2m
      labels:
        severity: critical
      annotations:
        summary: "Error rate > 5% for 2 minutes"
    
    - alert: HighP99Latency
      expr: |
        histogram_quantile(0.99,
          rate(http_server_requests_seconds_bucket[5m])
        ) > 2.0
      for: 5m
      annotations:
        summary: "P99 latency > 2s"
ELK Stack — centralized logging
Pod logs (stdout) ↓ Filebeat / Fluent Bit (DaemonSet — runs on every node) ↓ ships logs Logstash (parse, transform, enrich) ↓ Elasticsearch (store, index) ↓ Kibana (search, visualize, alert) Alternative (lighter): Loki + Promtail + Grafana Loki = "Prometheus for logs" — indexes only metadata, stores raw logs Cheaper than ES, native Grafana integration
# Fluent Bit DaemonSet — collects logs from all pods
apiVersion: v1
kind: ConfigMap
metadata:
  name: fluent-bit-config
data:
  fluent-bit.conf: |
    [INPUT]
        Name              tail
        Path              /var/log/containers/*.log
        Parser            docker
        Tag               kube.*
        Refresh_Interval  5
        Mem_Buf_Limit     50MB
    
    [FILTER]
        Name                kubernetes
        Match               kube.*
        Merge_Log           On        # parse JSON logs from Spring Boot
        K8S-Logging.Parser  On
    
    [OUTPUT]
        Name  es
        Match *
        Host  elasticsearch-master
        Port  9200
        Index hotel-service-logs-%Y.%m.%d

Distributed tracing — follow a request across services
When a user makes a booking: API Gateway → Booking Service → Payment Service → Notification Service. Without tracing you can't tell which service caused the latency. With OpenTelemetry, every request gets a trace ID that flows through all services. You see the full call chain and exact time spent in each service.
OpenTelemetry + Jaeger
Request enters API Gateway traceId = "abc-123" generated ↓ Booking Service (span 1: 150ms) └── DB query (span 2: 80ms) └── Payment Service call (span 3: 600ms) ← bottleneck! └── External gateway (span 4: 550ms) ← actual bottleneck └── Kafka publish (span 5: 10ms) ↓ Total: 760ms — Jaeger shows exactly where time went
# Spring Boot OpenTelemetry auto-instrumentation
# application.yml
management:
  tracing:
    sampling:
      probability: 0.1           # sample 10% of requests in production
  otlp:
    tracing:
      endpoint: http://jaeger-collector:4318/v1/traces

# Custom span — add business context
@Autowired Tracer tracer;

public Booking createBooking(CreateBookingRequest req) {
    Span span = tracer.nextSpan().name("createBooking").start();
    try (Tracer.SpanInScope ws = tracer.withSpan(span)) {
        span.tag("userId", req.getUserId());
        span.tag("roomId", req.getRoomId());
        return doCreateBooking(req);
    } catch (Exception e) {
        span.error(e);
        throw e;
    } finally {
        span.end();
    }
}
