MongoDB vs SQL — When to Choose Which
Beginner▼
AspectMongoDB (Document)PostgreSQL (Relational)
SchemaFlexible — each document can have different fieldsFixed schema — all rows have same columns
JoinsNo JOINs — embed or use $lookup (slow)Efficient JOINs with foreign keys
TransactionsACID since 4.0 (multi-doc)Full ACID with complex transactions
ScalingHorizontal sharding built-inVertical first, complex to shard
Best forCatalog data, CMS, logging, flexible schemas, hierarchical dataFinancial data, bookings, anything with complex relationships
Your UKG usePeople Fabric: WFM data with varying tenant configurationsBooking transactions at Agoda
Common Staff-level mistake: Choosing MongoDB because "it's flexible" and then doing complex aggregations that would be simple SQL JOINs. MongoDB's $lookup is much slower than SQL JOIN. If your data has many relationships — use PostgreSQL.

Schema Design — Embed vs Reference
The most important MongoDB design decision
Beginner▼
// EMBED — put related data inside the same document
// Use when: data always read together, child data is small, no sharing
{
  _id: "user-123",
  name: "Sonia Dua",
  email: "sonia@example.com",
  address: {                    // embedded — always read with user
    street: "123 Main St",
    city: "Noida",
    country: "India"
  },
  preferences: {               // embedded — small, user-specific
    currency: "USD",
    language: "en"
  }
}

// REFERENCE — store ID, fetch separately
// Use when: data is large, shared between documents, updated independently
{
  _id: "booking-456",
  userId: "user-123",           // reference to User collection
  hotelId: "hotel-789",         // reference to Hotel collection
  checkIn: "2026-07-01",
  checkOut: "2026-07-05",
  status: "CONFIRMED",
  payments: [                   // embedded — only belongs to this booking
    { amount: 500, currency: "USD", status: "PAID", paidAt: "..." }
  ]
}

// Decision rules:
// EMBED if: "Is the child data only meaningful in context of the parent?"
// EMBED if: "Is the child data small and bounded (won't grow unboundedly)?"
// REFERENCE if: "Is this document shared by multiple parents?"
// REFERENCE if: "Do I ever need to query the child independently?"

Aggregation Pipeline
The power tool — process and transform data server-side
Intermediate▼
// Aggregation pipeline — chain of stages, each transforms the data
// Like SQL: SELECT city, AVG(price) FROM bookings WHERE status='CONFIRMED'
//           GROUP BY city ORDER BY avgPrice DESC LIMIT 10
db.bookings.aggregate([
  { $match: { status: "CONFIRMED", checkIn: { $gte: ISODate("2026-01-01") } } },  // WHERE
  { $lookup: {                                   // JOIN
      from: "hotels",
      localField: "hotelId",
      foreignField: "_id",
      as: "hotel"
  }},
  { $unwind: "$hotel" },                         // flatten the array from $lookup
  { $group: {                                    // GROUP BY
      _id: "$hotel.city",
      avgPrice: { $avg: "$totalPrice" },
      totalBookings: { $sum: 1 },
      maxPrice: { $max: "$totalPrice" }
  }},
  { $sort: { avgPrice: -1 } },                  // ORDER BY avgPrice DESC
  { $limit: 10 },                               // LIMIT 10
  { $project: {                                  // SELECT specific fields
      city: "$_id",
      avgPrice: { $round: ["$avgPrice", 2] },
      totalBookings: 1,
      _id: 0
  }}
]);

// Common aggregation stages:
// $match    → filter documents (use early to reduce pipeline size)
// $group    → group + aggregate (sum, avg, min, max, count)
// $sort     → order results
// $limit    → limit results
// $project  → reshape documents (include/exclude/compute fields)
// $lookup   → left outer join with another collection
// $unwind   → deconstruct array field into multiple documents
// $addFields → add computed fields
// $facet    → multiple aggregations in one pipeline

Indexing — What Makes MongoDB Fast or Slow
Intermediate▼
// Without index: MongoDB scans EVERY document (collection scan) O(n)
// With index: jumps directly to matching documents O(log n)

// Single field index
db.bookings.createIndex({ userId: 1 })          // 1=ascending, -1=descending
// Now: db.bookings.find({userId: "u1"}) → fast

// Compound index — order matters!
db.bookings.createIndex({ userId: 1, status: 1, checkIn: -1 })
// Supports queries that use: userId alone, userId+status, userId+status+checkIn
// Does NOT support: status alone, checkIn alone (left prefix rule)

// Unique index — enforce uniqueness
db.users.createIndex({ email: 1 }, { unique: true })

// TTL index — auto-delete documents after N seconds (great for sessions, temp data)
db.sessions.createIndex({ createdAt: 1 }, { expireAfterSeconds: 86400 }) // 24h

// Text index — full text search
db.hotels.createIndex({ name: "text", description: "text" })
db.hotels.find({ $text: { $search: "luxury beachfront" } })

// Explain query — check if index is being used
db.bookings.find({ userId: "u1" }).explain("executionStats")
// Look for: "stage": "IXSCAN" (index scan = good) vs "COLLSCAN" (collection scan = bad)
// Also check: nReturned vs totalDocsExamined — big difference = missing index

// Spring Data MongoDB
@Document(collection = "bookings")
public class Booking {
    @Id private String id;
    @Indexed private String userId;                                 // single index
    @CompoundIndex(def = "{'userId':1, 'status':1, 'checkIn':-1}") // compound
    private String status;
}

Cosmos DB — Azure's Multi-Model Database
Your Walmart Secure Media Service used Cosmos DB — know the specifics
Advanced▼
Your Walmart context: Secure Media Service used Cosmos DB. When interviewers ask about Cosmos DB, they want to know: did you make design decisions (partition key, RU provisioning) or just query it? Be honest but show you understand the concepts.
// Cosmos DB key concepts
// 1. Partition key — most important design decision
//    All documents with same partition key = same physical partition
//    Queries within a partition = cheap (single partition query)
//    Queries across partitions = expensive (cross-partition query)

// Good partition key for media library: userId
// All media for a user accessed together, evenly distributed
{
  "id": "media-123",
  "userId": "user-456",        // PARTITION KEY — always include in queries
  "type": "IMAGE",
  "url": "https://cdn.example.com/img/123.jpg",
  "metadata": { "width": 1920, "height": 1080 }
}

// 2. Request Units (RU) — Cosmos DB's currency
// Every read/write costs RUs. Provisioned throughput = RU/s budget.
// Point read (by id + partition key): ~1 RU
// Query: varies (cross-partition = expensive)
// If you exceed RU budget: 429 Too Many Requests

// 3. Consistency levels (5 options — CAP trade-off)
// Strong:          all reads see latest write (like SQL). Highest latency.
// Bounded staleness: reads lag by X time/operations
// Session:         consistent within a session (default, good for most apps)
// Consistent prefix: reads never see out-of-order writes
// Eventual:        cheapest, lowest latency, may read stale data

// 4. Spring Data Cosmos DB
@Container(containerName = "media")
public class MediaDocument {
    @Id private String id;
    @PartitionKey private String userId;
    private String type, url;
}

public interface MediaRepository extends CosmosRepository<MediaDocument, String> {
    List<MediaDocument> findByUserId(String userId);  // single-partition query
}
