Authorization Code Flow with PKCE — step by step
1. User clicks "Login with Google" in your app 2. App generates code_verifier (random string) + code_challenge (SHA256 hash of verifier) 3. App redirects user to Auth0/Google authorization endpoint: GET https://auth0.com/authorize ?client_id=your-app-id &redirect_uri=https://yourapp.com/callback &response_type=code &scope=openid profile email &code_challenge=ABC123... ← PKCE &code_challenge_method=S256 &state=random-csrf-token 4. User sees login screen, enters credentials at Auth0/Google (never at your app) 5. Auth server redirects back to your app: GET https://yourapp.com/callback?code=AUTH_CODE&state=random-csrf-token 6. App verifies state (CSRF protection) 7. App exchanges code for tokens (server-side, never in browser): POST https://auth0.com/oauth/token code=AUTH_CODE code_verifier=original-verifier ← PKCE proof client_id=your-app-id client_secret=your-secret grant_type=authorization_code 8. Auth server validates code_verifier matches code_challenge 9. Auth server returns: { access_token: "JWT...", id_token: "JWT...", refresh_token: "opaque...", expires_in: 3600 } 10. App uses access_token to call your API Authorization: Bearer {access_token} 11. Your API validates token (signature + expiry + claims) on every request
TokenWhat it isWho uses itFormat
Access TokenAuthorizes API calls. Short-lived (15min-1hr). Contains user ID, roles, scopes.Your API validates it on every requestJWT (usually)
ID TokenIdentity information about the user. Returned by OIDC. Contains name, email, picture.Your frontend reads user info from itJWT always
Refresh TokenLong-lived (days/weeks). Used to get new access tokens without re-login. Store securely.Your app exchanges it for new access token when expiredOpaque string
Security rules: Access tokens → in memory only (not localStorage!). Refresh tokens → HttpOnly cookie or secure storage, never localStorage. ID tokens → decode for user info, never use to call APIs.

OAuth 2.0

Authorization framework. Answers: "What can this app do on your behalf?" Grants access to resources. Returns access tokens. Does NOT tell you WHO the user is.

OpenID Connect (OIDC)

Authentication layer on top of OAuth2. Answers: "Who is this user?" Adds ID token (JWT with user claims). Auth0, Google, Azure AD all implement OIDC. This is what you actually use for login.

Remember: OAuth2 = authorization ("can this app read your files?"). OIDC = authentication ("who are you?"). OIDC is OAuth2 + identity. Your Auth0 work at UKG implements OIDC for tenant authentication.

JWT — Structure, Validation, Vulnerabilities
Header.Payload.Signature — what each part is and how to validate safely
Beginner▼
# JWT = JSON Web Token = Base64URL(header).Base64URL(payload).Signature
# Example JWT (paste at jwt.io to decode):
# eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c2VyLTEyMyIsImVtYWlsIjoic29uaWFAZXhhbXBsZS5jb20iLCJyb2xlcyI6WyJBRE1JTiJdLCJpYXQiOjE3MTk5MTA4MDAsImV4cCI6MTcxOTkxNDQwMH0.signature

# Header (base64 decoded):
{ "alg": "RS256", "typ": "JWT", "kid": "key-id-1" }

# Payload (base64 decoded) — the claims:
{
  "sub": "user-123",          # subject (user ID)
  "email": "sonia@example.com",
  "roles": ["ADMIN", "USER"],
  "iss": "https://yourcompany.auth0.com",  # issuer
  "aud": "https://api.yourcompany.com",    # audience
  "iat": 1719910800,          # issued at (Unix timestamp)
  "exp": 1719914400,          # expires at (1 hour later)
  "jti": "unique-token-id"    # JWT ID (for revocation)
}

# Signature: RSASHA256(base64(header) + "." + base64(payload), privateKey)
# Validation: verify(base64(header) + "." + base64(payload), signature, publicKey)

--------------------------------------------------

# JWT validation — what your API MUST check (all of these)
1. Signature valid (use the PUBLIC key from JWKS endpoint)
2. exp not in the past (token not expired)
3. iss matches expected issuer (prevents tokens from other auth servers)
4. aud matches your API (prevents tokens meant for other services)
5. alg is expected algorithm (NEVER accept "none" algorithm!)

# Spring Boot JWT validation
# application.yml:
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://yourcompany.auth0.com/  # auto-fetches JWKS
          audiences: https://api.yourcompany.com

# JWT VULNERABILITIES — know these
# 1. alg=none attack: change algorithm to "none", remove signature
#    Fix: ALWAYS specify allowed algorithms, reject "none"
# 2. RS256→HS256 attack: server's public key used as HMAC secret
#    Fix: ALWAYS specify expected algorithm in validation
# 3. Expired token: exp not checked
#    Fix: always validate exp
# 4. JWT stored in localStorage: XSS can steal it
#    Fix: store access token in memory, refresh token in HttpOnly cookie

RBAC vs ABAC — Authorization Models
How to control who can do what in your system
Intermediate▼

RBAC — Role-Based Access Control

User has roles (ADMIN, USER, HOTEL_MANAGER). Permissions attached to roles, not users. Simple to understand. Good for: most applications with clear user types.

ABAC — Attribute-Based Access Control

Access decision based on attributes of user, resource, and environment. "User CAN edit hotel IF user.hotelId == hotel.id AND time is business hours." More flexible, more complex. Good for: fine-grained multi-tenant systems.

// RBAC in Spring Security
@PreAuthorize("hasRole('ADMIN')")
public void deleteHotel(Long id) { ... }

@PreAuthorize("hasAnyRole('ADMIN', 'HOTEL_MANAGER')")
public Hotel updateHotel(Long id, UpdateRequest req) { ... }

// ABAC — check attributes at runtime
@PreAuthorize("@hotelSecurityService.canEdit(authentication, #id)")
public Hotel updateHotel(Long id, UpdateRequest req) { ... }

@Service
public class HotelSecurityService {
    public boolean canEdit(Authentication auth, Long hotelId) {
        String userId = auth.getName();
        if (auth.getAuthorities().stream()
               .anyMatch(a -> a.getAuthority().equals("ROLE_ADMIN"))) {
            return true; // admins can edit any hotel
        }
        // Hotel managers can only edit their own hotel
        Hotel hotel = hotelRepo.findById(hotelId).orElseThrow();
        return hotel.getManagerId().equals(userId);
    }
}

API Security — Rate Limiting, CORS, Headers, Secrets
Advanced▼
# Security headers — what your API must set
Strict-Transport-Security: max-age=31536000; includeSubDomains  # force HTTPS
X-Content-Type-Options: nosniff                # prevent MIME type sniffing
X-Frame-Options: DENY                          # prevent clickjacking
Content-Security-Policy: default-src 'self'   # prevent XSS
X-XSS-Protection: 1; mode=block
Referrer-Policy: strict-origin-when-cross-origin

# Spring Boot: spring-boot-starter-security + Helmet (Node.js) add most of these

# CORS — Cross-Origin Resource Sharing
# Browser blocks JS from calling API on different domain by default
# CORS headers tell browser which origins are allowed

# Spring Boot CORS config
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("https://yourapp.com", "https://admin.yourapp.com")
            .allowedMethods("GET", "POST", "PUT", "DELETE")
            .allowedHeaders("Authorization", "Content-Type")
            .allowCredentials(true)
            .maxAge(3600);
    }
}
# NEVER: .allowedOrigins("*") with .allowCredentials(true) — security vulnerability

# Secrets management hierarchy

Local dev:     .env file (git-ignored) or docker-compose env
Staging/Prod:  
  Option A: GitHub Secrets (CI/CD env vars)
  Option B: AWS Secrets Manager / GCP Secret Manager (pull at runtime)
  Option C: HashiCorp Vault (enterprise, dynamic secrets)
  Option D: K8s Sealed Secrets (encrypted in git, decrypted by cluster)
  
NEVER:
  → Hardcode in code
  → Commit .env to git
  → Put in Docker image
  → Pass as plain CLI argument (visible in ps aux)
  → Store in localStorage or sessionStorage

# AWS Secrets Manager in Spring Boot
# application.yml:
spring:
  cloud:
    aws:
      secretsmanager:
        import-keys:
          - /hotel-service/prod/db-password
          - /hotel-service/prod/jwt-secret
# Secrets available as: ${/hotel-service/prod/db-password}
